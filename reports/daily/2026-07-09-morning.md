## Candidate papers/releases for `skill-bench`

### 1. SaaS-Bench: Can Computer-Use Agents Leverage Real-World SaaS to Solve Professional Workflows?

- ID: `arxiv:2605.15777v2`
- URL: https://arxiv.org/abs/2605.15777v2
- Extracted text: `data/papers/text/2605.15777v2-saas-bench-can-computer-use-agents-leverage-real-w.txt`
- Why it matters: This is directly in the “professional workflow benchmark” lane. The paper argues that existing web/GUI benchmarks overestimate agent ability because they have simplified app logic, isolated tasks, and short horizons; SaaS-Bench instead uses 23 deployable open-source SaaS systems across six professional domains, 106 cross-application tasks, multimodal evidence, and long-horizon workflows averaging over 100 interaction steps.
- Design transfer: The strongest reusable pattern is **real deployable systems + checkpoint-based partial credit**. Their task construction also uses a Builder-Challenger-Refiner process where LLM-generated candidates are iteratively reviewed for executability, verifiability, and professional realism. That maps well to a skill-bench pipeline where domain experts contribute traps/checkpoints rather than full benchmark implementations.
- Caveat: This is environment-heavy. The repo should borrow the verification/checkpoint and expert-review patterns before trying to replicate 23 SaaS systems.

### 2. MBABench: Evaluating LLM Agents on End-to-End Spreadsheet Tasks in Finance

- ID: `arxiv:2605.22664v4`
- URL: https://arxiv.org/abs/2605.22664v4
- Extracted text: `data/papers/text/2605.22664v4-mbabench-evaluating-llm-agents-on-end-to-end-sprea.txt`
- Why it matters: This is one of the clearest artifact-evaluation templates for skill-bench: complete multi-sheet financial workbooks rather than atomic formulas or spreadsheet Q&A. It explicitly frames professional quality as more than numeric correctness because workbooks are reviewed, modified, and reused by stakeholders.
- Design transfer: Their rubric decomposes workbook quality into **Accuracy** (50%; e.g. final calculations, starting values, task completion, sign consistency), **Formula** (35%; e.g. logic readability/size, edge cases, hardcoding, range hygiene, absolute references), and **Format** (15%; e.g. workbook structure, notation, readability, color standards, alignment, style consistency, presentation). This is a durable model for rubric design: separate answer correctness, maintainable artifact internals, and professional presentation.
- Caveat: Their quality criteria often require LLM-as-judge plus expert validation. For our public pilot, deterministic spreadsheet checks should cover the objective layer, with LLM/human judging reserved for maintainability and presentation.

### 3. OdysseyBench: Evaluating LLM Agents on Long-Horizon Complex Office Application Workflows

- ID: `arxiv:2508.09124v1`
- URL: https://arxiv.org/abs/2508.09124v1
- Extracted text: `data/papers/text/2508.09124v1-odysseybench-evaluating-llm-agents-on-long-horizon.txt`
- Why it matters: This is directly adjacent to AA-Briefcase-like office work: Word, Excel, PDF, Email, Calendar, long interaction histories, and cross-application reasoning. It includes 300 tasks transformed from real-world OfficeBench use cases and 302 newly generated tasks.
- Design transfer: HomerAgents is important less as an implementation target and more as a **task-scaling recipe**: transform atomic office tasks into contextually rich multi-interaction scenarios; separately synthesize new tasks via environment exploration, task generation, and dialogue creation. Their explicit finding that dialogue storage formats matter — semantic compression and coherent aggregation improve multi-step reasoning — is relevant for how skill-bench should package source corpora and prior-context histories.
- Caveat: Generated tasks need quality gates. For skill-bench, synthetic scenario expansion should be downstream of a small set of human-authored “gold” work products.

### 4. From Noisy Traces to Root Causes: Structural Trajectory Analysis and Causal Extraction for Agent Optimization

- ID: `arxiv:2607.07702v1`
- URL: https://arxiv.org/abs/2607.07702v1
- Extracted text: `data/papers/text/2607.07702v1-from-noisy-traces-to-root-causes-structural-trajec.txt`
- Why it matters: This is the most relevant newly discovered July 8 paper for the compounding-system side. STRACE treats long-horizon execution logs as structured causal evidence rather than monolithic text. It mines failure patterns across batches, filters redundant/low-value traces, performs backward causal localization over a textual execution dependency graph, and then updates the root-cause module’s instructions. They report a 42.5% → 58.5% improvement on VeruSAGE-Bench.
- Design transfer: Skill-bench should log failures in a way that supports later **root-cause slicing**: planner error vs evidence retrieval miss vs artifact-formatting miss vs tool-use/state-tracking miss. This matters for both agent optimization and benchmark diagnostics. If the benchmark only stores final artifacts and scores, it loses the signal needed to improve agents or skills.
- Caveat: The paper is optimization-oriented, not benchmark-design-first. Use it as a logging/diagnostics design pattern, not as a claim that automatic prompt repair will generalize to all skill-bench tasks.

### 5. Efficient Benchmarking of AI Agents

- ID: `arxiv:2603.23749v1`
- URL: https://arxiv.org/abs/2603.23749v1
- Extracted text: `data/papers/text/2603.23749v1-efficient-benchmarking-of-ai-agents.txt`
- Why it matters: Cost-aware evaluation is core to a compounding benchmark. The paper’s main result is that under scaffold and temporal shift, absolute score prediction degrades but rank-order prediction remains stable enough to reduce evaluation load.
- Design transfer: Their Mid-Range Difficulty Filter keeps tasks with historical pass rates between 30–70%, motivated by IRT, and reports 44–70% task reduction while preserving rank fidelity across eight benchmarks, 33 scaffolds, and 70+ model configurations. For skill-bench, this suggests a two-tier evaluation strategy: full runs for initialization/drift monitoring, reduced mid-difficulty suites for routine leaderboard or nightly agent comparisons.
- Caveat: The approach requires historical pass-rate matrices. It becomes useful after skill-bench has enough repeated runs, not on day one.

## Durable design pattern learned

A strong benchmark should separate three layers that are often blurred:

1. **Artifact quality**: Did the final spreadsheet/deck/memo satisfy professional criteria?
2. **Workflow state/checkpoints**: Did the agent make the right intermediate state changes and recover from errors?
3. **Trace diagnosis**: When it failed, which upstream module or knowledge primitive caused the failure?

MBABench is strongest on layer 1, SaaS-Bench on layer 2, and STRACE on layer 3. Skill-bench should intentionally design for all three from the beginning.
