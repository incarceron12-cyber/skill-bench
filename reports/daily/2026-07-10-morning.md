## Candidate papers/releases for `skill-bench`

### 1. LH-Bench: Skill-Grounded Evaluation of Long-Horizon Agents on Subjective Enterprise Tasks

- ID: `arxiv:2603.22744v2`
- URL: https://arxiv.org/abs/2603.22744v2
- Extracted text: `data/papers/text/2603.22744-lh-bench-skill-grounded-evaluation-of-long-horizon-agents.txt`
- Why it matters: This is the most directly relevant new fallback find. LH-Bench argues that long-horizon enterprise tasks cannot be judged by binary outcome checks alone because agents must satisfy procedural constraints like design-system discipline, source grounding, safe iterative editing, and intermediate artifact quality. Their key move is to use expert-authored `SKILL.md` artifacts as procedural evaluation knowledge: the same expert workflow expectations guide execution and define observable rubric boundaries afterward.
- Specific signal read: They instantiate this in Figma-to-code (33 real `.fig` tasks via Figma API/MCP) and programmatic content generation (183 chapters across 41 courses). Expert-authored skills improved LLM judge reliability over LLM-authored rubrics on the same runs (`κ=0.60` vs `0.46`), and structured verifier feedback enabled recovery from 70.3% of observed execution errors.
- Design transfer: Treat domain skills not merely as agent hints, but as **evaluation boundary objects**: each skill should encode workflow phases, common failure modes, completion criteria, and observable artifacts. This is exactly aligned with `skill-bench` as an expertise-to-evaluation engine.
- Caveat: The skill file may leak evaluation intent if published verbatim. For `skill-bench`, separate public procedural guidance from private artifact contracts / hidden rubric checks.

### 2. Workflow-GYM: Towards Long-Horizon Evaluation of Computer-use Agentic tasks in Real-World Professional Fields

- ID: `arxiv:2606.11042v3`
- URL: https://arxiv.org/abs/2606.11042v3
- Extracted text: `data/papers/text/2606.11042-workflow-gym-long-horizon-computer-use-agentic-tasks.txt`
- Why it matters: Workflow-GYM sits in the professional GUI/workflow lane: long-horizon tasks in configured environments with domain-specific professional software, natural-language task instructions, and deterministic final-state/artifact success criteria. The paper emphasizes that current GUI benchmarks over-index on simple/general-purpose apps and miss economically valuable professional work.
- Specific signal read: Strongest models achieve only slightly above 30% success; failure modes include workflow stage omission, error propagation, objective drift, repetitive action loops, and insufficient understanding of professional software environments.
- Design transfer: Its most reusable element is the **workflow abstraction**: a structured, goal-directed sequence of GUI actions that transforms an initial state into a final professional state or artifact. Even if `skill-bench` avoids heavy VM/GUI infrastructure initially, the same failure taxonomy should be used for office/file workflows.
- Caveat: Full configured professional software environments are infrastructure-heavy. Near-term transfer should be through failure-mode tags and final-state/artifact verification, not by attempting a complete GUI benchmark.

### 3. SaaS-Bench: Can Computer-Use Agents Leverage Real-World SaaS to Solve Professional Workflows?

- ID: `arxiv:2605.15777v2`
- URL: https://arxiv.org/abs/2605.15777v2
- Extracted text: `data/papers/text/2605.15777v2-saas-bench-can-computer-use-agents-leverage-real-w.txt`
- Why it matters: SaaS-Bench remains one of the clearest analogs for realistic professional workflow evaluation: 23 deployable SaaS systems, six professional domains, 106 long-horizon tasks, multimodal/text-only variants, and weighted verification checkpoints that measure both partial progress and strict completion.
- Specific signal read: The extracted text highlights that even the strongest model completes fewer than 4% of tasks end-to-end, while checkpoint scores expose partial capability. This is useful for `skill-bench`: long-horizon tasks should not collapse into a single fail bit.
- Design transfer: Borrow checkpoint scoring and state-based partial credit for tasks whose final artifact can fail despite meaningful intermediate progress.
- Caveat: The environment maintenance burden is too high for the first public pilot. Use as an evaluation pattern, not an implementation target.

### 4. MBABench: Evaluating LLM Agents on End-to-End Spreadsheet Tasks in Finance

- ID: `arxiv:2605.22664v4`
- URL: https://arxiv.org/abs/2605.22664v4
- Extracted text: `data/papers/text/2605.22664v4-mbabench-evaluating-llm-agents-on-end-to-end-sprea.txt`
- Why it matters: Still the strongest artifact-rubric template for `skill-bench`. It evaluates complete financial spreadsheets rather than atomic spreadsheet formulas, and explicitly treats professional quality as multi-dimensional because stakeholders must review and revise the workbook.
- Specific signal read: Their evaluation taxonomy separates **Accuracy**, **Formula**, and **Format**. That split maps cleanly to `skill-bench` artifacts: correctness, maintainable internal structure, and professional presentation.
- Design transfer: Build the first spreadsheet/memo pilot rubric with distinct objective checks, artifact-internals checks, and presentation/judgment checks. Do not let “right final number” dominate artifact usefulness.
- Caveat: Some maintainability/presentation checks need LLM or human judging; objective spreadsheet checks should anchor the public pilot.

### 5. OdysseyBench: Evaluating LLM Agents on Long-Horizon Complex Office Application Workflows

- ID: `arxiv:2508.09124v1`
- URL: https://arxiv.org/abs/2508.09124v1
- Extracted text: `data/papers/text/2508.09124v1-odysseybench-evaluating-llm-agents-on-long-horizon.txt`
- Why it matters: OdysseyBench directly targets office work across Word, Excel, PDF, Email, and Calendar. It uses 300 real-use-case-derived tasks and 302 newly synthesized tasks, each requiring agents to extract essential information from long-horizon interaction histories.
- Design transfer: The useful pattern is turning atomic office tasks into context-rich, cross-application scenarios. `skill-bench` should package source corpora and histories in ways that test context retrieval and aggregation, not just tool use.
- Caveat: Generated task expansion needs strong quality gates and human gold scenarios.

### 6. SciVisAgentBench: A Benchmark for Evaluating Scientific Data Analysis and Visualization Agents

- ID: `arxiv:2603.29139v2`
- URL: https://arxiv.org/abs/2603.29139v2
- Extracted text: `data/papers/text/2603.29139v2-scivisagentbench-a-benchmark-for-evaluating-scient.txt`
- Why it matters: Less central than office/SaaS/spreadsheet work, but strong on evaluation methodology for complex artifacts. It combines LLM-based judging with deterministic evaluators, including image metrics, code checkers, rule-based verifiers, and case-specific evaluators, plus expert agreement validation.
- Design transfer: Use a **hybrid evaluator stack** for subjective artifacts: deterministic checks where possible; judge checks for quality dimensions; expert validation to calibrate judge trust.
- Caveat: Domain is scientific visualization, so transfer the evaluation architecture rather than the task domain.

## Durable design pattern learned

Today’s new pattern is **skill-grounded evaluation**:

> Expert procedural knowledge should be captured as an intermediate artifact that can guide agents, constrain judges, define rubric boundaries, and diagnose failures — but the public guidance layer must be separated from private verifier contracts.

I updated `docs/state-of-the-art-map.md` with LH-Bench and Workflow-GYM entries. The broader architecture now looks like:

- MBABench: professional artifact quality.
- SaaS-Bench: checkpointed workflow state.
- LH-Bench: expert procedural rubrics / skills as evaluation boundary objects.
- Workflow-GYM: domain-specific GUI workflow realism and failure taxonomy.
- STRACE: causal trace diagnosis after failure.
