## Relevance, review status, charter fit, and decision filter

This is a **deep review** based on the complete immutable v1 primary source. It advances charter objectives A–C by testing a cross-domain question: when does a persistent workspace improve repeated knowledge work, and which evidence would distinguish retained configuration, executable artifact reuse, memory-mediated agent behavior, collaboration, and safe lifecycle operation? The review produces a durable treatment/claim audit and directly reusable measurement guidance. It is expansion and human learning, not a proposal to narrow `skill-bench` to dashboards or enterprise analytics. Useful completion means retaining the workspace-reuse mechanism while refusing unsupported general-memory and production claims.

## One-sentence contribution

The paper usefully packages user-authored specifications, profiled schemas, tool configuration, output contracts, executable artifacts, version history, and role-scoped workspace access into a reusable workspace, but its experiments do not isolate a selective-memory effect: the headline “memory” treatment bundles prompt content, prior artifact availability, interaction burden, and—in the public replication—replacement of agent generation with deterministic same-schema rerendering.

## Research question and intended construct

The paper asks whether recurring artifact work should preserve a curated workspace rather than reset every session or replay complete conversation history. It proposes four persistent categories (paper §§1, 3; pp. 1–4):

1. user-authored and refined task specifications;
2. automatically profiled data schemas;
3. tool/connector configurations and invocation templates;
4. output/runtime contracts, especially dynamic data injection rather than hardcoding.

The workspace also contains the latest generated artifact and its git history. At session start, task, tool, and output records are concatenated into the system prompt; the schema profile enters the user message. Conversation, reasoning, tool logs, temporary files, error recovery, raw data, and unapproved edits are omitted. Owners, stewards, and viewers receive different workspace permissions (pp. 4–5).

This is better described as **curated configuration plus artifact persistence** than as an autonomous memory extractor. The paper supplies no algorithm that discovers reusable content from a trace. Specifications are authored/refined by users, schemas are generated by fixed profiling, tool records come from platform configuration, and output constraints are system contracts. “Selective forgetting” is therefore a storage policy over known object classes—not learned selection, semantic conflict resolution, proven deletion, or demonstrated suppression of all alternate trace replicas.

## Methodology and system reconstruction

### Implemented configured system

The described deployment uses Claude Opus 4 for primary generation and Claude Sonnet 4 for schema validation/tool dispatch; a FastAPI backend; MongoDB workspace metadata; git artifact/version storage; CSV, SQL, REST, and MCP connectors; per-session file workspaces; and browser-rendered artifacts. Fresh sessions receive composed memory without being told that it is persistent (paper §§3–5, pp. 3–5). Exact model snapshots, prompts beyond one appendix example, SDK/package revisions, infrastructure, sampling settings, retry policy, and service dates are absent.

The zero-token path is a distinct mechanism. A previously generated program receives new data through a runtime injection point. Compatibility means only that every original column remains present; additional columns are allowed and missing columns trigger regeneration (p. 4). No LLM is called. This is executable artifact reuse under a shallow structural schema predicate, not memory retrieval or reasoning.

### Experiment 1: enterprise memory-condition ablation

The corpus contains 24 undisclosed enterprise files/tasks: 10 recurring refresh, 8 cross-team adaptation, and 6 iterative construction; data range from 200 rows/8 columns to 45,000 rows/42 columns (pp. 5–6). Each task is reported under:

- **no memory:** default prompt plus “full re-specification”;
- **full history:** complete prior transcript injection;
- **selective:** four structured categories **plus the prior artifact**, with traces omitted.

Two blinded raters judge render correctness, spot-checked data fidelity, format compliance, and completeness. Passing requires both raters to pass all four; κ = 0.91, but the κ type/unit, disagreement table, adjudication, rater training, and criterion definitions are absent. Means imply one result per task-condition: 23/24 selective, 19/24 no-memory, and 17/24 full-history; no stochastic repeats, missing/invalid-run accounting, task-level table, or output artifacts are supplied. Selective uses 3.4K input and 4.1K output tokens, 1.4 turns, and 68 seconds versus 2.1K/8.2K, 4.3 turns, and 285 seconds for no memory, and 18.7K/9.6K, 3.1 turns, and 310 seconds for full history (Table 1, p. 6).

The conditions are not content- or work-matched. No-memory explicitly requires re-specification, selective receives curated requirements and an artifact, and full history receives unspecified stale traces. The experiment estimates a **configured workspace package effect**, not selection versus retention in isolation.

### Experiment 2: public-data “replication”

Four named datasets—Superstore, UCI Adult, NYC 311, and World Bank GDP—receive three conditions and three runs each, for 36 stated trials (12 per condition). Automated checks require scripts, runtime injection, no hardcoded data, and dataset-specific content (pp. 6–7). The reported completions are 10/12 no-memory, 9/12 full-history, and 12/12 selective.

However, the selective arm is not an agent trial: it applies same-schema “V2” values to an already generated artifact, records zero input/output tokens, zero tool calls, zero seconds, and calls all 12 rerenders complete. No/full-history arms invoke an agent to generate artifacts. The source does not describe how V1/V2 forms were constructed, whether initial artifacts were selected after successful generation, whether all three conditions began from equivalent artifacts, or what changed besides values. The result supports a narrow software claim—twelve chosen artifacts rerendered on twelve schema-compatible updates—not a replicated selective-memory treatment or generalization claim.

### Experiment 3: token representation

Raw rows, first-50-row truncation, and a statistical summary are compared over the undisclosed enterprise data and four public datasets (pp. 6–7). Summary representation averages 0.5K tokens versus 48.7K enterprise and 473K public raw representations. This is a useful payload-size result. It does not show semantic sufficiency across joins, conditional distributions, rare values, time order, or task-specific calculations; indeed, the one selective enterprise failure is attributed to cross-file join semantics omitted from the summary, and truncation reportedly causes five enterprise errors.

Public token counts are estimated with OpenAI’s `cl100k_base` tokenizer although the evaluated agents are Claude models. They therefore are representation estimates, not demonstrated provider-billed input counts. No preprocessing time, profile bytes, storage, connector transfer, cache, initial artifact-generation cost, or amortization horizon is reported.

### Experiment 4: user study

Twelve participants (six engineers, six analysts) perform four “counterbalanced tasks.” Shared memory is reported as 14× faster for recurring generation (12 versus 165 seconds), 2.5× faster for refinement, and 3× faster for constrained generation; “would use again” is 6.5 versus 4.2/7, and revert is used 1.8 times per session (p. 7). The source does not state recruitment, prior product familiarity, allocation sequence, comparator definitions, task instances, whether each participant saw every condition, learning/carryover washout, exclusions, per-condition denominators, dispersion, participant-level results, or statistical models. It explicitly calls the study underpowered. It is formative usability evidence, not collaboration utility, population preference, or productivity-effect validation.

## Evidence: what is and is not supported

The enterprise observations are consistent with the proposition that supplying concise, accepted configuration and prior artifact state can reduce turns and improve rubric passing relative to forcing users to restate requirements or injecting a long transcript. The reported full-history failures also motivate stale-context tests. The paper does **not** expose task-level paired outcomes, transcript composition, failure labels, or artifacts, so the mechanism cannot be independently checked.

There is also a serious statistical reporting problem. Recomputing ordinary two-sided Fisher exact tests from the only integer counts compatible with the rounded completion rates gives **p = 0.1882** for selective (23/24) versus no-memory (19/24), not the reported 0.046, and **p = 0.0479** for selective versus full-history (17/24), not 0.008. Because the same 24 tasks appear in each condition, an appropriate paired test would require the unreported discordance table; it cannot be reconstructed from marginals. The paper neither identifies paired versus independent Fisher testing nor supplies task-level outcomes. Consequently, neither reported significance claim is auditable from v1.

The 12/12 zero-token observation has no uncertainty analysis and is conditioned on schema compatibility and an existing artifact. Twelve successes do not establish refresh reliability, and “100% completion” collapses transport, parse, transformation, semantic correctness, render correctness, and changed-data consequence into an unpublished automated verdict.

## Unique insight: workspace reuse needs a multi-link claim ladder

The genuinely useful insight is not “selective memory beats history.” It is that repeated artifact work can reuse **different durable objects through different execution paths**. A specification can enter a prompt; a schema can guide code generation; a connector can authorize data access; an artifact can execute without a model; and a version snapshot can be restored without generation. Calling all of these memory obscures what succeeded.

For `skill-bench`, preserve this chain:

`retained object and authority → permitted accessor → retrieval/composition → model-visible evidence (if any) → semantic adoption/rejection → action or artifact execution → observed state/artifact consequence → criterion result`.

The zero-token route legitimately skips model visibility and adoption:

`versioned artifact + compatible runtime data → executed transformation → rendered/structured state → semantic consequence checks`.

That is a valuable architectural branch, but it cannot be counted as a successful memory-mediated agent trial. Likewise, a role bit permits access but does not establish that the object’s originating authority, purpose, data rights, or decision mandate transfers to another user.

## Limitations and validity threats

### Construct and treatment validity

1. **No extraction/selection treatment.** The categories are manually designed and populated by users, deterministic profiling, or platform configuration. There is no candidate trace, selector precision/recall, rejected-item audit, or category ablation.
2. **Selective is a package, not one factor.** It bundles curated prompt material, schema summary, tool setup, output contract, prior artifact, versioned state, and lower user burden. Full history and no-memory do not receive matched information and budgets.
3. **Artifact reuse is mislabeled as memory completion.** Public selective trials perform no agent generation and begin with reusable artifacts; they cannot estimate a memory effect against generation arms.
4. **No stale/conflicting selective-memory test.** Staleness is inferred from full transcript behavior. Selective records are not experimentally contradicted, superseded, revoked, partially applicable, or placed under schema/credential/tool drift. Memory decay is future work.
5. **Schema compatibility is syntactic.** Column-subset acceptance misses type changes, units, keys, cardinality, nullability, category drift, temporal semantics, join relations, distribution shifts, and changed business definitions.
6. **“Full history” is under-specified.** Transcript length, exact content, artifact inclusion, ordering, truncation, current-query placement, and parity with selective information are absent. “Trace anchoring” is a post-hoc label with no coding protocol or intervention.

### Measurement and statistics

7. **Single stochastic enterprise runs.** Twenty-four tasks appear to receive one run per condition under mutable proprietary models. No seeds, temperatures, retries, invalids, or repeated-run variance are reported.
8. **Reported Fisher tests are not reproducible.** Marginal-count recomputation disagrees, while the paired discordance needed for a matched test is unavailable.
9. **Rubric evidence is unavailable.** Four criterion names, κ = 0.91, and sparse failure summaries do not expose rater evidence views, alternate-valid artifacts, criterion dependence, or adjudication.
10. **Public checks are unreleased.** “Dataset-specific content verification” cannot be audited for semantic coverage, alternate outputs, reward hacking, or data leakage.
11. **Means hide dependence and missingness.** Tasks cluster in three use-case families and likely share templates, data lineages, raters, and platform behavior. There are no confidence intervals or family-cluster estimates.
12. **User-study denominators are unclear.** Counterbalancing does not cure practice, product-familiarity, or task carryover without disclosed assignment and analysis.

### Sharing, governance, and safety

13. **Access control is described, not evaluated.** No authorization matrix test, confused-deputy case, credential isolation audit, cross-workspace canary, revocation test, or viewer-to-steward escalation test is reported.
14. **Sharing permission is not semantic authority.** A colleague’s task rules, thresholds, tool permissions, and output conventions may be inappropriate for a new region, role, dataset, or affected party even when technically readable.
15. **Deletion/privacy claims are too broad.** The paper says no PII is stored and traces are not persisted, but provides no data inventory, log/cache/backup audit, retention/deletion test, connector secret-flow evidence, or independent security assessment. Schema samples, category catalogs, prompts, artifacts, URLs, and credentials can themselves be sensitive.
16. **Revert is not memory correction.** Git restoration demonstrates artifact-state recovery if implementation behaves as described; it does not revoke stale prompt records, erase replicas, repair shared derivatives, or prove downstream consumers stopped using superseded state.

### Reproducibility and operational realism

There is no release, frozen task corpus, V1/V2 public data pack, prompt set, transcript set, task-level outcomes, rater sheet, user-study instrument, automated grader, output artifact, trial trace, environment image, dependency lock, model snapshot, usage ledger, or analysis code. The enterprise corpus is private and only aggregate means are published. Even the named public replication cannot be reproduced from the paper because task instructions, initial artifacts, V2 construction, checks, and exact system are absent.

The platform shape is operationally plausible—versioned artifacts, connectors, role-scoped workspaces, drafts, and runtime refresh are real production concerns—but deployment is asserted rather than characterized. No user/account count, observation period, workload sampling frame, incident/missing-run denominator, service availability, latency distribution, dollar cost, maintenance burden, security audit, or production outcome is given. The evidence supports an experience report plus small authored evaluations, not production reliability or broad enterprise effectiveness.

## Transfer to `skill-bench`

### Retain

1. Model persistent configuration, schema/evidence summaries, tool capabilities, output contracts, executable artifacts, versions, and access policy as separate typed objects.
2. Record accepted versus draft/unapproved state and preserve immutable parent/child version lineage.
3. Test both model-mediated reuse and zero-model deterministic rerun; report them as different execution modes and estimands.
4. Include full-history and no-history controls, but equalize information, artifact availability, work budget, and current-task requirements before attributing a selective-memory effect.
5. Treat summaries as lossy transformations with source locators, transformation hashes, omitted-information declarations, and task-specific sufficiency checks.

### Repair and test

For a future cross-domain persistent-workspace slice, use matched `reset / full-history / curated-correct / curated-stale / curated-conflicting / curated-revoked / artifact-rerun` conditions. Version and hash every retained object and composition. Plant type/unit/key/category/join/semantic schema changes, not just missing columns. Observe access, presented evidence, adoption/rejection, action, artifact/state delta, collateral effects, and final checks separately. Add role/authority transfer, revocation, derived-copy, and rollback cases. Repeat stochastic agent arms and cluster inference by task lineage; treat deterministic reruns as software conformance trials with their own compatibility and semantic-oracle coverage.

### Do not infer

Do not infer general memory capability, learned selection, safe forgetting, causal trace harm, cross-user authority, collaboration benefit, privacy compliance, professional competence, production reliability, or deployment readiness from these results. Do not combine a zero-call artifact rerender and an agent generation into one completion-rate estimand.

## Concrete repository actions

No new queue task is warranted. The existing benchmark-bundle traces, persistent-workspace/evidence-chain records, artifact-view admissibility, compounding lessons, longitudinal reset/persistence protocol, expert-participation authority lineage, task-health, metric, and validity contracts already cover the nonduplicate obligations. The next relevant builder should refine an existing memory/workspace pilot rather than create another schema:

1. add the matched stale/conflict/revocation/artifact-rerun conditions and the two execution-mode estimands above;
2. replace column-subset “compatibility” with typed semantic compatibility and consequence checks;
3. require role permission and originating semantic authority to be observed separately.

## Bottom line

The paper offers a credible product-design hypothesis: curated workspace state and executable artifacts can spare users repeated specification and avoid replaying noisy transcripts. Its strongest reusable contribution is the separation of prompt-composed configuration from zero-model artifact refresh. But v1 does not identify a selective-memory effect cleanly, does not reproduce its reported significance from published counts, and provides no inspectable release. The public 12/12 result is a same-schema rerender conformance observation, not twelve successful memory-agent trials. `skill-bench` should retain the versioned workspace architecture while instrumenting every authority, composition, adoption, execution, and consequence link—and should keep agent-mediated memory utility distinct from deterministic artifact reuse.
