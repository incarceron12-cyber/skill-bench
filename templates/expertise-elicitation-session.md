# Expertise elicitation session record

> **Purpose:** generate provenance-bearing candidate benchmark primitives from a
> real contribution. This is an authoring instrument, not evidence that any claim
> is universally true or release-ready. Preserve exact responses in a linked
> transcript/recording when consent permits; do not rewrite contributor statements
> silently.

### Evidence labels used throughout

Tag every entry with one or more:

- `SPONTANEOUS`: contributor supplied it before the relevant category/probe.
- `PROBED`: elicited by a recorded interviewer prompt.
- `CONTRIBUTOR_INFERENCE`: contributor interpretation or counterfactual.
- `ANALYST_INFERENCE`: analyst mapping; not contributor testimony.
- `OBSERVED`: supported by direct workplace/artifact observation.
- `DOCUMENTED`: supported by a cited source.
- `CORROBORATED`: independently supported; name the evidence.
- `CONTRADICTED`: conflicting evidence exists; do not resolve silently.

### Scope questions

- Who consumes the work product, and what decision does it support?
- Which role actually owns the decision?
- What forms/artifacts/states must be produced?
- Which tools, policies, safety/compliance rules, and time constraints govern it?
- Which contexts, organizations, jurisdictions, customer types, or technologies
  materially change the work?
- What would useful completion of this elicitation look like?

## 2. Task diagram: locate expertise hotspots

Ask: **“Think about what you do when you [task]. Can you break it into three to
six broad stages?”** Keep this high-level; it is a roadmap, not a complete process.

| Stage | Inputs/artifacts | Output/state | Cognitively demanding? Why? | Consequence of error | Evidence label + locator |
|---|---|---|---|---|---|
| | | | | | |

If one stage is too broad, decompose only that stage once. Mark the stages selected
for deepening and why.

## 3. Knowledge audit: examples, not maxims

For each selected hotspot, request a **specific lived example**. Use only relevant
probes and record the exact prompt:

- past/future projection;
- big-picture integration and competing goals;
- cues or patterns that “pop out”;
- job smarts and efficient resource use;
- improvisation or an opportunity;
- self-monitoring and adjustment;
- anomaly or absence of an expected event;
- misleading tool, data, or guidance;
- a standard procedure that needed contextual adaptation.

| Claim/example ID | Hotspot | Exact probe | Concrete incident/example | Cues and evidence used | Strategy/threshold | Why difficult / predicted novice error | Scope and exceptions | Evidence label + locator |
|---|---|---|---|---|---|---|---|---|
| | | | | | | | | |

**Follow-ups for every example:**

1. How did you know? What observable cues or sources mattered?
2. What alternatives did you consider or reject, and why?
3. What would change your assessment or action?
4. In what context would this advice be wrong?
5. Is this approved practice, a personal preference, or a workaround?
6. What artifact/state would show that the work was done well or badly?
7. Who else or what source could corroborate or challenge it?

## 4. Critical incident / simulation deepening

Select a consequential incident in which the contributor made a key decision. It
must come from lived experience unless explicitly labeled `SIMULATED`. Avoid an
incident memorable only for reasons unrelated to the contributor's judgment.

### 4.4 Deepening and counterfactuals

For each turning point, ask only relevant probes:

- What cues were present, absent, ambiguous, or contradictory?
- What knowledge or analogue gave those cues meaning?
- What goals and stakeholder consequences mattered?
- What options were available; what was rejected and why?
- What threshold triggered action, escalation, or verification?
- What error might a less-experienced person make, and why?
- What if a key feature changed? What stays invariant?
- What if the data/tool/procedure were misleading or unavailable?

| Turning point | Cue/evidence predicates | Goal/threshold | Rejected alternative | Counterfactual variation | Predicted failure signature | Scope/caveat | Locator |
|---|---|---|---|---|---|---|---|
| | | | | | | | |

## Method provenance

Adapted for skill-bench from the complete primary-source review in
`docs/concepts/cognitive-task-analysis-to-benchmark-authoring.md`, principally:

- Militello et al. (1997), ACTA Task Diagram, Knowledge Audit, Simulation
  Interview, Cognitive Demands Table, and evaluation revisions;
- Hoffman et al. (2008), document bootstrapping, proficiency scaling, workplace
  analysis, Critical Decision Method, coding verification, and Decision
  Requirements Table.

Project additions include evidence-status labels, transformation lineage,
public/private fairness, contradiction preservation, benchmark-utility review,
and yield metrics. These additions are not claims made or validated by the CTA
reports.
