   The evaluation process itself remains fundamentally manual and labor-intensive. Reviewers
follow author-provided instructions to verify artifacts can execute and reproduce claimed results,
earning badges based on demonstrated capabilities (Figure 1). Recent studies reveal an uncomfort-
able truth: successful evaluations often depend less on documented procedures than on reviewers’
ad hoc debugging and environment repair [27]. This reliance on implicit expert knowledge sig-
nificantly undermines both efficiency and consistency. Hermann et al. report that the absence of
explicit quality standards leaves reviewers without guidance on acceptance thresholds, leading to
inconsistent evaluations across reviewers, conferences, and years [21].
   Artifact evaluation automation appears promising with the development of large language mod-
els (LLMs) [20]. Recent advances in LLM-driven agents have demonstrated impressive capabilities
across software engineering tasks, from bug localization to automatic repair [33, 37, 42]. Further-
more, recent advances in repository-level environment setup and testing, such as [22] and [13],
enable automated project building and execution. However, these agents target specific software
engineering tasks and do not align well with the unique characteristics of artifact evaluation. Unlike
other SE tasks with unified specifications and clear success criteria, artifact evaluation confronts
diverse repository structures, instructions targeting paper reproduction rather than general execu-
tion [7], and explicit verification against badge criteria [43]. To our knowledge, there have been no
prior works on end-to-end automation of artifact evaluation.
   We begin by asking: what makes artifact evaluation resistant to automation? To answer
this question, we have constructed a dataset from the field of software engineering, which includes
48 research papers and artifacts from the top four software engineering conferences. We then
conducted a preliminary study on a quarter of the dataset and identified three representative
challenges hindering artifact evaluation automation. First, perceiving execution states is hindered

by the lack of structured representation; documentation intermixes executable commands with
narrative text, making it difficult to model and track the evaluation progress. Second, maintaining
stable execution environments is complicated by dynamic boundary shifts; a single docker run
command can transparently relocate operations from host to container, yet documentation rarely
makes this transition explicit. Third, recovering from execution errors is hindered by failures that
can arise throughout the workflow, from environment construction to command execution.
   These findings drive the design of ArtifactCopilot, the first agent-based framework to au-
tomate end-to-end artifact evaluation from PDF input to badge determination. Our core insight
is treating evaluation as a state-aware workflow problem rather than simple script generation.
To address the first challenge, we propose the AE Graph, which transforms unstructured doc-
umentation into a structured state representation, enabling precise modeling of the evaluation
progress. To tackle the second challenge, we introduce an environment-aware execution monitor
that dynamically tracks context switches between host and container, ensuring commands are
dispatched to the correct runtime. For the third challenge, we develop an agent framework for
automated error recovery spanning both environment construction and artifact evaluation phases.
   Evaluated on 48 real-world artifacts, ArtifactCopilot produces evaluation outcomes that
match human-assigned artifact badges in 85.42% of cases, representing a substantial improvement
over LLM-based script generation (20.83%) and Claude Code (33.33%), while incurring minimal
human effort (0.11 interventions/artifact) and cost ($0.091/artifact). Ablation studies confirm the
indispensability of key modules, showing that omitting the AE Graph, environment unification,
or multi-agent collaboration, respectively, reduces success rates by 43.75%, 29.17%, and 22.92%. In
summary, our contributions are as follows:
• A new perspective on artifact evaluation automation. We conceptualize artifact evaluation
  as a state-aware workflow problem rather than a one-shot script generation task, revealing
  fundamental challenges in execution state perception, environment boundary management, and
  error recovery that are not addressed by existing automation approaches.
• An analytical foundation for understanding artifact evaluation. We construct a real-world
  dataset of 48 artifact evaluations collected from top-tier software engineering conferences and
  conduct a systematic preliminary study, providing the first empirical characterization of obstacles
  that make artifact evaluation resistant to automation.
• The first end-to-end automated artifact evaluation framework. We propose Artifact-
  Copilot, an agent-based framework that transforms unstructured PDF documentation into a
  structured execution model and automates artifact evaluation from instruction parsing to badge

  multi-agent error recovery.
• Strong empirical results on real-world artifacts. Evaluated on 48 real-world artifacts, Ar-
  tifactCopilot achieves an 85.42% agreement with human artifact evaluation outcomes, out-
  performing LLM-based script generation and existing coding agents, while requiring minimal
  human intervention (0.11 interventions per artifact) and low monetary cost ($0.091 per artifact).

2   Dataset Construction
Motivated by the challenges of automating artifact evaluation and the need for systematic assess-
ment of automation tools, we prioritized the construction of a comprehensive dataset. To the best
of our knowledge, there has been no prior research on constructing datasets for automated artifact
evaluation. [7] is limited to artifacts utilizing LLM API calls; these rely on commercial APIs and lack
the universality required for general artifact evaluation. Consequently, we constructed a dataset
specifically designed to benchmark automated techniques.

   The dataset construction workflow is illustrated in Figure 2. We sourced research papers from the
ACM Digital Library [10], targeting four top-tier software engineering conferences—ICSE, FSE, ASE,
and ISSTA. To ensure alignment with contemporary artifact evaluation standards, we prioritized
conferences from 2024 onwards in which AE processes were concluded, and badge information
was publicly available. The selected venues include ISSTA 2024 [2], ASE 2024 [3], ICSE 2024 [4],
and ICSE 2025 [6]. Additionally, as the AE results for FSE 2024 were undisclosed at the time of data
collection, FSE 2023 [1] was included to guarantee comprehensive coverage across all four major
venues. From these conferences, we identified all papers awarded either the Artifacts Functional or
Artifacts Reusable badge. From the above pool, we employed simple random sampling to select 100
candidate papers. This sample size was determined to balance statistical representativeness with
the feasibility of rigorous manual verification. At the same time, random sampling allows us to
maintain an unbiased selection and ensures that each artifact receives an in-depth human review.
   We applied exclusion criteria to isolate intrinsic reproducibility challenges from issues of external
accessibility. Artifacts requiring commercial software (30), private datasets (9), specialized hardware
(7), or excessive disk space exceeding 10 GB (6) were excluded, with some artifacts meeting multiple
criteria, resulting in 48 retained artifacts. To verify the exclusion decisions, two authors jointly
spent approximately 60 hours reviewing the 100 candidates, reflecting the substantial manual
effort required and justifying the necessity of our sampling strategy. We further categorized
the retained artifacts by environment management, identifying 25 artifacts using Docker-based
containerization and 23 relying on non-containerized setups (e.g., Conda).
   During dataset construction, we observed substantial heterogeneity in artifact evaluation criteria
across venues and years (e.g., ICSE 2024 [23] omitted the Artifacts Functional badge, whereas ICSE
2025 [5] included it). To ensure a unified evaluation standard aligned with ACM guidelines [9], we
normalize all evaluation outcomes to the Artifacts Functional level. As the Artifacts Reusable badge
is awarded only when the functional criteria are satisfied and relies on subjective assessments
of documentation quality, it is excluded from metric computation; artifacts awarded the Artifacts
Reusable badge are thus treated as functionally positive.
   To operationalize functional correctness, we construct a Golden Command Set (GCS) for each
artifact by manually extracting all essential execution commands from the official README files.
Each README is independently reviewed by two annotators, with disagreements resolved by a
third researcher, yielding a validated command set that serves as the ground-truth specification for
functional execution.
   Together, these design choices normalize the dataset to a unified functional evaluation space, in
which each artifact is associated with (i) a binary functional label derived from official evaluation
results and (ii) a corresponding GCS defining its expected execution behavior. This normalization
provides a consistent foundation for the metric definitions in § 5.1

3     Understanding Automation Barriers: A Preliminary Study
Before conducting the study, we defined the scope of end-to-end artifact evaluation as spanning from
the completion of repository download to the completion of the artifact badge evaluation process;
link acquisition and repository download are excluded as preparatory steps. The path to automation
requires understanding what humans actually do during evaluation and why these actions resist
mechanization. We conduct a preliminary study to answer the following two questions:
• RQ1: What operations require human intervention during artifact evaluation, and how frequently
  do they occur?
• RQ2: What technical obstacles hinder automated artifact evaluation, and where do they originate?
We partitioned the artifacts into an exploration set (12 artifacts) and a validation set (36 artifacts)
using stratified random sampling by execution environment type. The exploration set was used
to inform empirical analysis and system design decisions, whereas the validation set was held
out until final evaluation. The exploration set supports qualitative analysis rather than statistical
generalization. To mitigate individual cognitive bias, two authors with over three years of software
engineering experience independently executed each exploration artifact while adhering strictly to
the README instructions.
   During execution, we recorded all steps at which the workflow would become blocked and
unable to proceed without human input or modification, which we define as human intervention
steps. We were permitted only minimal adjustments, including correcting evident typographical
errors, installing undocumented dependencies, or resolving version conflicts. When Docker usage
was specified, we directly pulled the provided image or built the environment from the supplied
Dockerfile; otherwise, we executed the artifact using a Dockerfile configured with an Ubuntu 22.04
LTS base image. All artifacts were executed on a uniform server configuration (Ubuntu 22.04 LTS,
NVIDIA RTX 4090, Docker 24.0.1) to minimize environmental variability.

performed collaboratively by two authors to ensure consistency, while subsequent classification
followed open coding [19] with independent labeling, achieving Cohen’s 𝜅 = 0.85 across three
categories, as demonstrated in Table 2, which represents high consistency. Figure 3 illustrates some
cases arising during manual artifact evaluation that need human intervention.

3.2    RQ2: Structural Barriers to Automation
RQ1 quantifies human effort by identifying where and how often human intervention occurs
during artifact evaluation. RQ2 builds directly on these observations by analyzing the underlying
technical and structural reasons that give rise to the intervention scenarios identified in RQ1.
Beyond observable interventions lies a subtler question: what implicit knowledge enables humans
to navigate evaluation successfully, and why is this knowledge difficult to automate? We identified
two structural barriers that demand such tacit understanding.
   Implicit task state maintenance. Human reviewers maintain a holistic mental model of the
evaluation process, including execution history, current state, and intended goals. Our analysis
corroborates this requirement: across all 12 artifacts, reviewers repeatedly backtracked to verify
progress, and in 10 cases consulted files beyond the README to resolve state ambiguities. This global
awareness allows reviewers to interpret underspecified instructions, assess the impact of errors,
and adjust execution paths accordingly. By contrast, documentation encodes task dependencies
implicitly in natural language, interleaving commands with prose; tools that treat READMEs as
linear command sequences therefore lack explicit task structure and cannot maintain a coherent
execution context. This barrier explains a substantial portion of the command execution and error
handling interventions observed in RQ1, where reviewers must decide not only which command to
execute next, but whether prior steps need to be revised or repeated after failures.
   Execution environment switching. Beyond cognitive challenges, artifacts introduce sub-
stantial technical complexity that further impedes automation. Half of the artifacts (6 of 12) rely
on interactive Docker containers invoked via docker run -it, which fundamentally shift the
execution context from the host to the container, altering filesystem roots, environment variables,
and available tooling. Such context boundaries are rarely made explicit in documentation, as hu-
man reviewers are expected to recognize and adapt to them implicitly. While humans naturally
accommodate this transition, automated tools that execute commands sequentially may operate
in an incorrect namespace or block indefinitely, awaiting container termination, which is largely
invisible in intervention frequency statistics (RQ1).

3.3      Implications for Design
The empirical findings crystallize three challenges for end-to-end automation, and these challenges
directly inform the design of ArtifactCopilot, as detailed in the next section.
Challenge 1 (C1): perceiving execution states. As revealed by RQ2, human success depends on
maintaining a global task context that remains implicit in artifact documentation. Automated tools,
therefore, require explicit representations encoding execution phases, inter-command dependencies,
and artifact relationships to enable context-aware reasoning and error impact analysis.
Challenge 2 (C2): maintaining stable execution environments. Also identified in RQ2, inter-
active containers and heterogeneous specifications create invisible context boundaries. Tools must
detect environment transitions, manage container lifecycles, and normalize diverse configurations
into executable environments.
Challenge 3 (C3): automating execution and recovering from execution errors. Grounded in
RQ1, command execution and error handling together account for nearly 93% of interventions. An
automated system must parse documentation, extract commands, detect failures, and attempt repairs
without human involvement. Additionally, it must resolve placeholders and handle interactive
prompts to prevent execution from stalling.

embedded in linear scripts. Based on this formulation, ArtifactCopilot integrates graph-based
task modeling with a hierarchical agent pipeline, where a Planning Agent materializes execution

4.1      Overview and Architecture
ArtifactCopilot accepts two input modes: papers in PDF format or pre-downloaded repositories
with confirmed availability. In both cases, the system produces structured evaluation reports with
badge recommendations. Figure 4 illustrates the end-to-end workflow for the PDF-based setting,
organized into three stages.
Stage 1: Repository acquisition and task modeling primarily addresses C1 by identifying
artifact links in papers, retrieving corresponding repositories, and constructing the AE Graph, a
structured representation that translates README instructions into an explicit, dependency-aware
workflow. This stage makes execution states and task relationships explicit, forming the basis for
downstream reasoning.
Stage 2: Environment preparation mainly targets C2. It employs a Planning Agent to analyze
dependencies, construct containerized execution environments, and unify the execution environ-
ment to host from the command level. Ultimately, it outputs a set of meta-task specifications that
constrain subsequent execution.
Stage 3: Execution and evaluation focuses on C3, while also handling residual issues from
earlier stages. An Evaluation Agent executes commands following the AE Graph topology,
detects and recovers from runtime errors, and generates structured evaluation reports with badge
recommendations, enabling robust and resilient artifact evaluation.

4.2   Explicit Task Modeling via AE Graphs
4.2.1 Paper Parsing and Repository Acquisition. ACM Artifact Available badges require artifacts
to be publicly accessible and verifiably associated with the paper [23]. To establish a reliable
foundation for subsequent task modeling, ArtifactCopilot automates repository acquisition in
three steps. (1) Link extraction: an LLM analyzes the paper PDF, focusing on experimental and
data availability sections, to identify repository URLs via keyword-guided parsing, and produces a
concise summary of the evaluation setup for later correspondence checking. (2) Repository download:
ArtifactCopilot normalizes extracted URLs, resolves redirections across hosting platforms (e.g.,
GitHub [17], Zenodo [15]), and retrieves the referenced repositories. (3) Correspondence verification:
semantic similarity between the paper-derived summary and the repository README is used to
confirm that the artifact corresponds to the paper. If any step fails, the system reports diagnostic
information and halts automated evaluation, and the artifact is treated as unavailable.

4.2.2 From Documentation to AE Graph. As discussed in § 3.2, humans implicitly maintain struc-
tured task representations capturing execution phases, dependencies, and artifact relationships.
Providing raw README files to agents, however, leads to two limitations: execution context grows
as commands accumulate, and non-sequential dependencies remain implicit, hindering failure
analysis and recovery. Inspired by control-flow and data-flow graphs in program analysis [28, 36],
we observe that artifact evaluation is driven not by variable states but by command execution states.
Task progress depends on which commands succeed and how commands and artifacts depend on
one another. We therefore introduce the AE Graph, a directed graph 𝐺 = (𝑉 , 𝐸) in which nodes
represent commands and artifacts, and edges encode execution and artifact dependencies. Table 3
summarizes the formal definition.
   AE Graph construction proceeds in two phases. First, ArtifactCopilot aggregates evaluation-
relevant documentation, including README, REQUIREMENTS, and SETUP files. Second, an LLM
parses these guidelines into an initial AE Graph using prompt engineering with Chain-of-Thought
reasoning [34]. The resulting graph enables agents to (i) query global task state at any execution
point by tracing dependencies, allowing rapid identification of affected downstream steps when
failures occur, and (ii) maintain a compact execution context by storing state in node attributes
rather than verbose logs. Prior work shows that such structured representations improve LLM
reasoning over unstructured text [14]. Figure 5 illustrates an example transformation from README
to AE Graph. The initial graph is subsequently validated and incrementally refined in later stages.

4.3        Planning Agent: Environment Normalization and Planning
4.3.1 Design Rationale. Before command execution, preparing an isolated environment is a primary
prerequisite. As shown in § 3.2, artifacts exhibit substantial heterogeneity not only in dependencies
but also in execution context specification: some provide Dockerfiles, others rely on ad-hoc runtime
instructions, while many implicitly assume system-level configurations. Human reviewers naturally
interpret these context switches; automated evaluation requires making them explicit. We therefore
introduce a dedicated Planning Agent to normalize execution semantics before evaluation. Given
artifact-specific environment specifications, the Planning Agent constructs a corresponding
Docker environment and externalizes the execution context to the host. All commands are issued
from the host and redirected to the target container through a fixed execution interface. This
design eliminates implicit host–container transitions and allows the Evaluation Agent to execute
workflows directly from the host without entering containers.

4.3.2 Adaptive Container Construction. The Planning Agent first validates artifact nodes by
traversing the AE Graph and verifying file paths against the repository structure. Path mismatches
trigger name-based search and substitution, while genuinely missing artifacts are recorded for
downstream handling. To enable uniform host-driven execution, the Planning Agent constructs a
container substrate using three adaptive strategies based on artifact-provided configurations:
Strategy 1: Dockerfile reuse. When artifacts provide Dockerfiles or pre-built images, the agent
extracts the base image and entrypoint definitions, and builds or pulls the corresponding container.
Strategy 2: Automatic Dockerfile synthesis. When artifacts provide only dependency mani-
fests (e.g., requirements.txt), the agent infers system and runtime dependencies, synthesizes a
Dockerfile, and builds the container image.
Strategy 3: Template fallback. When dependencies are incomplete or image construction fails
after 𝑘=3 attempts, Planning Agent falls back to a predefined Ubuntu 22.04 base template.
   This adaptive construction step prepares a controllable container substrate for subsequent
execution context normalization, while accommodating the configuration heterogeneity observed
in § 3.2. Figure 6 illustrates an example Dockerfile automatically generated by the Planning Agent.

4.3.3 Execution Environment Normalization. The observation in § 3.2, a number of artifacts rely on
interactive container commands (e.g., docker run -it), introducing implicit execution context
switches between host and container. While human evaluators naturally track these transitions,
automated tools executing commands may apply subsequent instructions in incorrect namespaces.

   The Planning Agent enforces a host-centric unified scheduling model in which all commands are
issued from the host and explicitly mapped to target containers through a fixed execution interface.
The system leverages container execution APIs for host-driven command injection. Containers
with customized entrypoints, however, exhibit execution behaviors that are incompatible with a
single mechanism. Accordingly, a dual strategy is employed. Containers with standard entrypoints
accept commands injected directly through container execution interfaces with container-aware
path mappings. Containers with custom entrypoints are initialized with a detached shell session, in
which the original entrypoint is replayed as the first instruction, followed by sequential command
injection. This design ensures that command execution consistently remains on the host, without
requiring entry into the Docker container.

4.3.4 Meta-Task Specification. The final output of the Planning Agent is a structured meta-task
specification that formalizes the evaluation procedure for each artifact. The specification includes
repository context, agent role, and an ordered task decomposition into four semantic stages: Check
(artifact completeness validation), Run (command sequence execution), Verify (output compari-
son against paper claims), and Evaluate (badge eligibility determination). Each specification is
instantiated per artifact and encoded in a machine-readable format for the Evaluation Agent. An
example meta-task specification is shown in Figure 6.
   In addition to the meta-task specification, the Planning Agent outputs: (1) an updated AE Graph
with validated artifacts, container-adapted paths, and execution annotations; and (2) a container
environment description with identifiers, execution modes, and path mappings. Together, these
artifacts constitute the Evaluation Agent’s initial execution context, enabling direct workflow
execution under normalized execution semantics without re-parsing documentation.

4.4      Evaluation Agent: Execution and Badge Determination
4.4.1 Design Rationale. The Evaluation Agent operates exclusively on structured inputs pro-
duced by the Planning Agent, including the AE Graph, Docker information, and meta-task
specifications. Its responsibility is limited to execution and result assessment under normalized
execution semantics. The evaluation workflow consists of three stages: graph-guided command
execution, failure recovery, and badge determination. By design, environment construction and
execution context normalization are excluded from this agent, allowing it to focus on execution
logic and outcome analysis.

4.4.2 Graph-Guided Execution with Recovery. The Evaluation Agent executes commands by
traversing the AE Graph in topological order while maintaining execution states at the node
level. Compared to linear execution, this structure provides two advantages. First, upon failure,
dependency relations encoded in the graph allow selective continuation by isolating affected
downstream nodes rather than terminating the entire workflow. Second, node-level status tracking
serves as compact execution checkpoints, reducing reliance on verbose execution logs.
   For recovery, the Evaluation Agent analyzes failure traces using an LLM-based diagnostic
module and generates targeted repair actions. Each command is retried up to five times before
being marked as failed, after which downstream tasks continue independently. Beyond explicit
failures, the agent detects implicit blocking through lightweight resource monitoring. Sustained
low CPU utilization across intervals is interpreted as stalled execution, triggering log analysis and
recovery. This mechanism addresses dynamic interactive execution behaviors identified in § 3.2.
4.4.3 Badge Determination and Reporting. Upon completion of graph execution, the Evaluation
Agent produces two outputs. The first consists of detailed execution logs recording command
statuses, outputs, and errors for auditability. The second is an evaluation report summarizing
execution outcomes, error analyses, and comparisons between observed results and expected
claims extracted from the abstract of the paper.
   Badge determination follows ACM artifact evaluation criteria [9]. The Artifact Functional badge
requires successful execution of all graph nodes with expected outputs. Assessment of the Artifact
Reusable badge is derived from evidence collected during execution and reported in a standardized
format for reviewer inspection.

4.5    Implementation Details
Both agents follow the ReAct paradigm [38], in which the LLM processes static prompts together
with accumulated execution history and produces structured (Thought, Tool Call, Arguments) ac-
tions at each step. The agents share a common toolset consisting of five primitives: read_file for file
inspection and content extraction, write_file for persisting generated content, command_execute
for host-level shell execution, docker_check for retrieving container metadata, and task_finish
for terminating the ArtifactCopilot workflow.
   Context management is handled through a hierarchical strategy that preserves critical execution
states under constrained context windows. At the step level, verbose build logs and program
outputs are filtered using salience-aware truncation. Empirically, essential signals such as command
invocations and failure indicators concentrate near the beginning and end of outputs; accordingly,
head–tail truncation is applied when outputs exceed a fixed threshold. At the session level, long-
running evaluations are managed through semantic compression. When the accumulated context
approaches the model limit (e.g., 100k tokens), historical execution traces are summarized into a
compact state description that replaces raw logs as the new context baseline.
   System stability and efficiency are further controlled through execution safeguards and prompt
layout optimization. A strict upper bound on interaction turns prevents unbounded feedback loops.
In addition, the prompt template is organized to exploit context caching mechanisms provided
by modern LLM APIs: static instructions and task descriptions precede dynamically appended
execution history, improving cache reuse across successive calls. An example of the resulting
prompt structure is shown in Figure 7. Complete prompts and configuration details are included in
the replication artifact (see the Data Availability section).

6     Discussion
6.1     Failure Analysis and Limitations
A root-cause analysis was conducted on the seven cases that failed to complete the full AE workflow
to characterize the capability boundaries of ArtifactCopilot. The failure categories and their
distributions are summarized in Table 6.
Environment and dependency issues (4 cases, 57.1%) represent the dominant failure source and
primarily arise during environment construction. These failures are typically caused by implicit
system-level dependencies, such as undocumented Python library incompatibilities or unavailable
toolchains in containerized settings. While the Planning Agent employs dependency-driven
synthesis and fallback strategies, it relies on declared configurations and lacks the ability to infer
implicit dependencies from source code. As a result, repeated failures exceed the retry threshold.
Data dependency issues (2 cases, 28.6%) occur during instruction execution and manifest as
missing or empty intermediate data artifacts. Although documentation specifies data acquisition

steps, the Execute Agent lacks an external context for validating data completeness or correctness.
This limitation reveals that the current AE Graph captures artifact existence but does not model

Configuration mismatch (1 case, 14.3%) results from implicit cross-file constraints, such as
inconsistent path assumptions across scripts. Because the AE Graph is derived from document-level
semantics, such constraints remain unmodeled. This indicates that documentation analysis alone is
insufficient for capturing certain execution assumptions.
   Overall, these failures reflect the combined effects of system limitations and incomplete artifact
specifications. When dependencies and data requirements are explicitly declared, ArtifactCopilot
can reliably complete artifact evaluation. However, missing or implicit assumptions substantially
reduce the effectiveness of automated recovery. The current evaluation scope of ArtifactCopilot
is limited to artifacts that meet predefined resource and accessibility constraints. Artifacts that
require specialized hardware, private datasets, or large-scale computational resources remain
outside the supported scope. Addressing these limitations will require tighter integration of code-
level dependency analysis, stronger data flow validation, and finer-grained constraint modeling
within the AE Graph.

6.2   Implications
Based on both the empirical analysis of real-world artifact evaluation outcomes and hands-on
experience gained during the design and implementation of ArtifactCopilot, we distill the fol-
lowing insights. These implications are grounded in observed failure patterns, human intervention
behaviors, and ablation results, and aim to inform future standardization efforts, author practices,
and the design of automated Artifact Evaluation systems.
Implications for standardizing the AE process. The analysis in § 3.1 reveals substantial het-
erogeneity in artifact configurations and documentation quality. These observations suggest that
the artifact evaluation process benefits from being structured into clearly verifiable stages with
explicit input-output specifications. The stages include artifact integrity checking, environment
configuration and build, instruction execution and error recovery, and badge level assessment. For
artifacts that depend on non-standard resources, such as specialized hardware or private datasets,
explicit and structured declarations of such constraints are necessary in order to avoid implicit
assumptions during evaluation.
Recommendations for artifact authors. As 53.16% of manual interventions involve basic com-
mand execution according to § 3.1, many artifacts lack automated validation entry points. Several
practices can substantially reduce evaluation friction. These include providing one-click validation
scripts preferably via containerized setups, separating lightweight validation workflows from full
reproduction pipelines, and externalizing configuration parameters such as API keys and dataset
paths rather than hard-coding them. The failure analysis in § 6.1 indicates that failures stem from
undocumented assumptions, including undeclared dependencies and inconsistent configurations.
Implications for automated tool design. The ablation study in § 5.3 indicates that structured
task modeling plays a central role in automated artifact evaluation. Removing the AE Graph leads
to substantial performance degradation. These findings reinforce the importance of representing
evaluation workflows using explicit dependency structures rather than linear instruction sequences.
They also highlight the need to decouple environment construction from execution and verification.
Explicit modeling of execution boundaries in containerized settings is necessary to mitigate failures
caused by implicit context switching.

6.3   Threats to Validity
Internal validity. Threats to internal validity mainly arise from three factors. First, LLM outputs
are inherently stochastic, and identical inputs may yield different execution paths. This effect was
mitigated by setting the model temperature to zero and enforcing structured output constraints at
key decision points. Second, the retry threshold of the error recovery mechanism (𝑘 = 3) and the
blocking detection interval of three minutes were selected based on pilot experiments. Alternative
parameter settings may alter the trade-off between success rate and resource consumption. Third,
the classification of human intervention scenarios in § 3.1 was conducted using an open coding ap-
proach and involved approximately ten person-hours of manual analysis. Although inter-annotator
agreement was high with a Cohen’s 𝜅 value of 0.85, some subjectivity may remain.
External validity. Threats to external validity primarily concern dataset representativeness. The
evaluation dataset was collected from artifacts published at four top-tier software engineering
conferences, namely ICSE, FSE, ASE, and ISSTA. Artifacts that require specialized hardware, private
datasets, or large-scale computational resources were excluded. Consequently, the findings may not
directly generalize to other research domains or resource-intensive artifacts. Nevertheless, software
engineering artifacts exhibit substantial diversity and complexity. The core observations regarding
human intervention patterns and automation bottlenecks are therefore expected to provide useful
insights for artifact evaluation practices beyond the evaluated scope.

7    Related Work
Artifact Evaluation. With the increasing emphasis on open science and research reproducibility,
artifact evaluation has been widely adopted by top-tier conferences in software engineering,
computer systems, and artificial intelligence as a mechanism for encouraging the release of research
artifacts. A growing body of work has examined the effectiveness and limitations of AE from
multiple perspectives.
   At the macro level, empirical studies by Liu et al. [26] and Childers and Chrysanthis [16] report
substantial growth in artifact availability and show that papers passing AE often exhibit advantages
in citation impact. However, these studies also identify persistent challenges, including insufficient
artifact maintenance and highly variable documentation quality. From the perspective of repro-
ducibility quality, large-scale investigations by Muttakin et al. [27] and Siddiq et al. [32] highlight
a pronounced gap between artifact availability, executability, and reproducibility. Their findings
indicate that practical obstacles such as dependency management, environment configuration, and
access restrictions remain widespread, even among artifacts that have received AE badges.
   At the methodological and practice-oriented level, Gundersen and Kjensmo [18] and Semmelrock
et al. [31] introduce multi-level reproducibility frameworks and barrier-focused analyses that
provide conceptual foundations for understanding AE outcomes. Complementing these efforts,
Saucez et al. [30], Beyer and Winter [11], and Runeson et al. [29] summarize AE organization
processes, evaluation experiences, and community-specific best practices. While these studies have
significantly advanced institutional design, evaluation criteria, and empirical understanding of AE,
artifact evaluation is generally treated as a predominantly manual activity.
   Systematic investigations of automated support for Artifact Evaluation remain limited. In particu-
lar, existing work has not yet examined how automation can be used to support artifact acquisition,
environment construction, instruction execution, and result verification in an integrated manner.
Reducing evaluation costs while improving consistency and scalability. therefore remains an open
challenge. Unlike prior AE studies that focus on policy, process, or empirical outcomes, this work
studies artifact evaluation as an end-to-end automation problem.
LLM-based Agents. Recent advances in large language models have given rise to agent-based
approaches with strong automation capabilities in software engineering. Prior work has applied
LLM agents to diverse tasks, including code generation and translation, build configuration, test
execution, and automated program repair.
   For code understanding and transformation, TRANSAGENT by Yuan et al. [40] adopts a multi-
agent collaboration paradigm that combines control-flow–driven code alignment with execution-
level semantic alignment, enabling fine-grained localization and repair in cross-language code
translation. CODEAGENT by Zhang et al. [41] integrates external programming tools with multi-
strategy agents to support information retrieval, code navigation, and execution verification in real
repositories, improving the reliability of repository-level code generation.
   Several studies further demonstrate the effectiveness of LLM agents in software building, execu-
tion, and repair. CXXCrafter by Yu et al. [39] analyzes dependencies and build configurations and
performs iterative repairs from error feedback to improve build success for complex C/C++ projects.
Repo2Run by Hu et al. [22] addresses environment configuration using a dual-environment design
with atomic configuration synthesis to improve the runnability of Python repositories. ExecutionA-
gent [13] and RepairAgent [12] by Bouzenia and Pradel target build/test execution and automated
program repair, simulating human debugging via meta-prompting, tool use, and explicit state
modeling. At the system level, SWE-agent [37] introduces an Agent Computer Interface (ACI)
for interacting with real development environments and outperforms prior methods on multiple

8   Conclusion
This paper presents ArtifactCopilot, the first end-to-end agent-based framework for automat-
ing software artifact evaluation. Based on an empirical study of real-world artifacts, we identify
three key challenges: perceiving execution states from unstructured documentation, handling
implicit execution environment switches, and recovering from pervasive execution failures. Ar-
tifactCopilot addresses these challenges by formulating artifact evaluation as a state-aware
workflow problem, using the AE Graph to model commands, artifacts, and dependencies, and a
hierarchical architecture that separates environment planning from execution and verification
under normalized, host-centric execution semantics. Evaluated on 48 artifacts from top-tier soft-
ware engineering conferences, ArtifactCopilot achieves an 85.42% BCR with near-zero human
intervention, substantially outperforming representative LLM-based baselines.
