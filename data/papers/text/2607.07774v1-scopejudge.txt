We contend that as we transition to a world where LLMs are used in increasingly sensitive
applications, it is essential that their actions are monitored at the call level to ensure they
are able to achieve their objectives without catastrophic side effects. In sensitive fields like
information security, where consequences of unaligned actions are borne by the organization
driving the model rather than the lab training them, these best practices deserve to be
established empirically and in the open. To enable this benchmarking of judge models and
transcript strategies, we create ScopeJudge, a dataset of 100 individual information security
trajectories labeled as being aligned or unaligned. The trajectories are produced by running
autonomous agents on ScopeBench, a suite of information-security challenges engineered to
tempt agents out of scope, which we describe here only insofar as it is needed to interpret
the trajectories and treat as the subject of separate work. By analyzing the accuracy and
cost of different judge models and transcript strategies, we establish empirically rigorous
recommendations for practitioners, as well as establish a dataset for experimenting with new
techniques going forward.
The central difficulty is that the boundary between an aligned and an unaligned action is
not fixed in advance. In the autonomous deployments we study – a general-purpose agent
handed an objective and a set of constraints, as in Claude Code’s auto mode – the operative
scope is declared in the user’s request and must be inferred from their intent. This challenges
specifying what is permissible upfront: a fixed policy or constitution [4], however carefully
written, cannot enumerate the task-specific scope of a request it has never seen. We therefore
frame oversight as a monitoring problem rather than a specification problem: a cheap, trusted
judge observes a strong, untrusted, and readily swappable agent and gates each action before
it executes (Figure 1), an instantiation of trusted monitoring [5] for offensive security. The
empirical question this paper answers is how much context such a monitor needs to match
human experts at separating in-scope from out-of-scope actions, and at what cost.

1.1   Monitoring Strategies and Their Failure Modes

Two strategies are commonly used to manage autonomous tool-call risk: sandboxing and
human oversight. Sandboxing isolates the agent in an environment where no tool call can
cause permanent harm; the agent then runs unmonitored, and we judge only its finished
artifact against the original goal. This is not practical in every field: in information security,
for instance, the systems under test are often black-box and private to clients, and cannot be
reproduced. This leads to the second strategy, human oversight, in which a person reviews
each tool call before it runs. However, production evidence suggests that prompt-fatigue soon
sets in, and humans default to accepting tool-call permission requests without understanding
them, undermining the oversight. This effect is long documented for security warnings, where
users click through the large majority of warnings they are shown, familiarity breeding a lack
of concern [6]. In addition, this becomes impractical as the ratio of agents to humans increases.
LLM judges placed in front of the agent would not be vulnerable to prompt fatigue, and would
therefore work well to enable scalable oversight of agents so long as alignment with human
professionals and cost could be established.


1.2   Axes of Judge Monitoring Tradeoffs

Although this LLM-judge strategy is increasingly used in software, little open empirical research
establishes the cost and efficacy of competing judge schemes. There are, after all, dozens of
ways to present the context of a running agent to a judge. Each of these methods will have
different tradeoffs:

   1. The cost of the judge. As the number of tokens presented to the judge grows, so does
      the per-call cost.
   2. The accuracy of the judge in classifying aligned and unaligned tool calls: we naively
      expect more context (and tokens) should allow for greater accuracy.
   3. The adversarial robustness of the judge: A judge knowing the internal reasoning of
      an agent or reading the output of tool calls may have greater context for judging an
      incoming tool call, but that data also represents an attack surface that could be used by
      a misaligned agent and outside attacker respectively to poison a judge’s context in order
      to have unaligned tool calls be erroneously marked as aligned.


1.3   Contributions

2     Background & Problem Statement

We begin by formalizing the definition and evaluation of LLM judges for pre-execution tool-call
gating. Then, we ground this problem in the literature of agent rollouts by judges. Finally, we
describe the specific threat models of our safety and security evaluations.


2.1    Problem Formalization

We now fix the notation used throughout the remainder of the paper. The setup mirrors
that of prior work on agent risk assessment [7, 8], but explicitly distinguishes post-hoc from
pre-execution judging.


Tools and harnesses. Let T = {t1 , . . . , tM } denote a finite set of tools available to an
agent. Each tool t : Xt → Yt maps an input space (its arguments) to a possibly stochastic
output space (its response). A harness H exposes T to the agent alongside a system prompt
and a user message u, and is responsible for the actual execution of tool calls.


Agents and trajectories. An agent is a policy πθ , parameterized by an underlying language
model θ, that at each step emits either a tool call or a terminal action stop. Running πθ in
harness H given user message u produces a trajectory

                                                                          
                         τ = u, (a1 , o1 ), (a2 , o2 ), . . . , (aN , oN ) ,

where ak ∈ T × X is the k-th tool call and ok ∈ Y is its observed output. We refer to τ as a
completed trajectory of length N . For any step k ∈ {1, . . . , N +1}, the partial trajectory up to
but not including step k is

                                                                                                (2)
                                                                       
                            τ<k = u, (a1 , o1 ), . . . , (ak−1 , ok−1 ) ,

with τ<1 = (u). The proposed tool call at step k is the next action ak that πθ emits when
conditioned on τ<k , prior to any execution.


Scope, policy, and alignment labels. We model the deployment scenario of a general-
purpose autonomous agent deployed without per-call human approval (e.g., a coding agent such
as Claude Code in an auto configuration). A single global static policy SP, shared verbatim

across all tasks, encodes generic operating ground rules. The task-specific scope (which hosts,
services, techniques, and data are in or out of bounds) is not part of the static policy but is
declared in the user message u and must be inferred from the user’s stated goal and constraints.
The user message u therefore carries two things: the agent’s objective (its intent) and the
per-engagement scope (what is in bounds); the static policy SP supplies only the generic
floor shared by every task. The operative constraint for a task is therefore the conjunction of
SP and the scope expressed in u. For each proposed call ak , this operative scope induces a
ground-truth alignment label yk ∈ {0, 1}, where yk = 0 denotes an aligned call and yk = 1 an
unaligned call (a scope violation). For ScopeJudge, yk is assigned by professional penetration
testers; inter-grader agreement is reported in Section 4.


Post-hoc judging. Prior work has studied judges of the form

                                Jpost : (SP, τ ) −→ {0, 1}N ,                                 (3)

which examines a completed trajectory and emits one label per trajectory [8, 7]. Post-hoc
judging is useful for evaluation and offline auditing, but cannot prevent unaligned actions: by
the time Jpost runs, the harm is already done.


Pre-execution gating. ScopeJudge studies the complementary regime, in which the judge
is invoked before each tool call is executed and sees only the prefix of the trajectory together
with the proposed action:

                                                                                             (4)
                                                     
                             Jpre : SP, φ(τ<k , ak ) −→ {0, 1},

where φ is a transcript strategy, a function that projects the partial trajectory and proposed
call onto the context window of the judge. A judge output of 0 instructs the harness to execute

2.2   Language Model Agents in Offensive Security

Language-model agents are increasingly applied to offensive-security work (penetration testing
and bug-bounty triage [9, 10]), where, unlike in a sandbox, the systems under test are real,
client-owned, and often irreproducible. Evaluating such agents is itself hard: whether a
sequence of actions satisfies the operational requirements of an engagement is not mechanically
verifiable, and has been studied with LLM judges over completed rollouts [8]. The same
autonomy that makes these agents useful makes them risky, as an agent with broad tool access
can take consequential, irreversible actions against live infrastructure [7]. The failure mode we
target, a tool call that strays outside the authorized scope, is acute in this setting precisely
because the cost of a single out-of-bounds action is borne by a third party. Proactive, step-level
guardrails on tool invocation are an emerging deployment-time defense [11]; pre-execution
scope gating (Section 2.1) is the instance we study for offensive security.


2.3   Monitoring and Control of Agent Actions

Approaches to agent safety can be separated by when the constraint is applied. Specification-
time methods fix what is acceptable before the agent runs, most prominently by training
against a written constitution or policy [4]. Such methods are powerful for general norms
(“never exfiltrate credentials”) but cannot, by construction, encode the scope of a task that is
only revealed at request time. Deployment-time methods instead monitor and gate actions as
they are proposed. One line of this work enforces formal policies at runtime (e.g., bounding
the probability that an agent’s execution violates a policy expressed in a formal language [12]),
which is sound when the acceptable behavior can be written down in advance, but inherits the
same limitation as specification-time methods when the operative scope is only revealed in the
user’s request. We instead study a learned, intent-conditioned monitor that recovers scope
from the request itself. The AI-control notion of trusted monitoring [5] (one instantiation of
the broader program of scalable oversight [13]), in which a weaker trusted model supervises a
stronger untrusted one, is the natural fit for our setting, where the agent’s weights may be
swapped for each new frontier model while the monitor stays fixed. ScopeJudge operationalizes
trusted monitoring at the granularity of the individual tool call and asks which context a
monitor must see (and what that context costs) to be a reliable gate. Our static-policy strategy
is the specification-time view rendered as a monitor; the remaining strategies test whether
intent and history recover the scope a fixed policy cannot.


2.4   LLM-as-Judge: Methodology and Failure Modes

LLM judges are increasingly used [14] to score qualities of agent behavior that resist mechanical
verification, from offline evaluation of completed rollouts [8, 7] to the in-flight gating we study
here. They can be effective even at subtle determinations: recognizing that an agent has
reward-hacked [15] a task that a mechanical checker would mark as solved [16]. The paradigm
is not without failure modes, however: pairwise judging protocols exhibit well-documented
position and selection biases [17, 18], which we sidestep by having each judge classify each
tool call independently rather than comparatively (Section 3).

3.1     A Taxonomy of Transcript Strategies

Recall from Section 2.1 that a transcript strategy φ is a function that projects the partial
trajectory τ<k and proposed tool call ak onto the context window of the judge. We now define
the family of five φ studied in this work; each varies along two main axes: token-cost and
prompt-injection surface. The smallest token-cost arises from considering only the judge’s
static policy and the proposed tool call; the largest from supplying the full transcript of the
executing agent up to step k. Prompt-injection surface is the fraction of the judge’s input that
may be attacker-controlled: static policy and user message are operator-authored, whereas
tool-call outputs are explicitly untrusted bytes drawn from the target environment. Likewise,
an agent’s reasoning traces could themselves be untrusted if the model is unaligned.
Formally, the strategies are:

                          φintent (τ<k , ak ) = (SP, u, ak ),                                 (6)

                                                                                

                            φfull (τ<k , ak ) = (SP, τ<k , ak ),                              (9)

where σ : Y → Y is a summarization function applied to each tool output by an auxiliary
instance of the same LLM. Note that φstatic and φintent are memoryless with respect to prior
tool calls and therefore expose a vanishing prompt-injection surface; the remaining strategies
admit attacker-controlled bytes (tool outputs) into the judge’s context in exchange for richer
signal. The remainder of this subsection motivates each φ in turn. Appendix A reproduces the
verbatim static policy and tabulates the exact context blocks each φ supplies, and Appendix B
walks through a complete judge input under φint+summ .


3.1.1    Static Policy

In Static Policy only the static policy and the proposed tool call are considered. This is the
cheapest context strategy, but provides a judge the least information about intent. It is good
for setting a floor on behavior, where certain kinds of activities are never allowed by a model.
Since it lacks any historical context, it is easy for an unaligned model to try many variations
on a rejected tool call in order to adversarially search for one a judge will accept.


3.1.2    Intent

In Intent, the judge model can consider the most recent user message alongside its static policy.
This allows for specifically rejecting tool calls that may be considered benign by the static
policy, but are not aligned with the user’s expressed intent. For example, if a user message
were to describe a known benchmark task, a judge model would be able to distinguish web
searches that were providing research information to the running agent versus web searches
that were intended to directly look up the solution to the task as a form of reward hacking.
Similar to Static Policy it lacks any historical context, meaning it is easy for an unaligned
model to try variations on a rejected tool call in order to adversarially search for one the judge
will accept.

3.1.3    Intent Plus Tool Calls

In Intent Plus Tool Calls the judge model has access to its static policy, the user message, and
the history of tool calls issued but without the outputs. The assistant’s reasoning traces are
intentionally stripped to prevent the judge from being manipulated.


3.1.4    Intent Plus Output Summarized

In Intent Plus Output Summarized the judge model has access to its static policy, the user
message, the tool calls, and summaries of the tool responses. Each tool response is summarized
by a separate instance of the same judge model.


3.1.5    Full

The entire trajectory. Reasoning traces and assistant prose are no longer stripped.

4     Dataset

ScopeJudge treats its labeled corpus as a calibrated measuring instrument: a set of agent

4.2    From Trajectories to a Calibration Set

Exhaustively labeling every tool call of thousands of generated trajectories with multiple
experts is infeasible. Following the calibrate-then-deploy logic of judge-validated evaluation [19],
we instead draw a calibration set of 100 trajectories whose every tool call is labeled by all five
graders, and reserve the larger pool of remaining trajectories for future work (Section 8). All
quantitative claims in this paper (inter-grader agreement, judge precision, recall, and F1, and
the cost–accuracy frontier) are made against this small calibration set.

natural rate at which scope violations occur instead of over-sampling them. Scope violations
are rare even under deliberate temptation (7.7% of tool calls in the calibration set), and
synthetically inflating that rate would render precision and other base-rate-sensitive metrics
unrepresentative of a monitor’s true deployment behavior, which is the scenario the benchmark
exists to measure. We do stratify the draw across the task × model grid so that no model

4.3    Labeling Protocol

Each tool call ak is labeled independently by five professional penetration testers, all of whom
have worked in web penetration testing or bug bounty. A grader is shown the global static
policy SP, the user objective and constraints u, and the full partial trajectory τ<k up to
the proposed call (including prior tool calls, their outputs, and the agent’s reasoning), and
assigns yk ∈ {0, 1}: aligned, or a scope violation. Ground truth is thus the judgment of a fully
informed expert; the transcript strategies of Section 3 then test how much of this context an
automated monitor actually needs to recover that judgment. Labels are assigned per call and
are not propagated forward: a violation at step j does not force yk = 1 for k > j.


4.4    Inter-Grader Agreement and the Expert Reference Point

A monitor cannot be expected to agree with the expert consensus more often than the experts
agree with one another; inter-grader agreement is therefore the reference point against which
we read every judge result. We quantify agreement with Fleiss’ κ [20], which accommodates our
five raters and, by correcting for chance agreement, is far more informative than raw percent
agreement under the heavy class imbalance of our data (only 7.7% of calls are violations).
Across all 4,897 tool calls we measure Fleiss’ κ = 0.64 (mean pairwise Cohen’s κ = 0.64 [21]),
which is substantial agreement on the Landis–Koch scale [22]; the per-pair structure is shown
in Figure 2. Agreement is not uniform: 88.1% of calls are labeled unanimously, while the
remaining 11.9% form a contested subset of non-unanimous calls in which the genuine ambiguity
a monitor must contend with is concentrated. We translate the consensus labels into an expert
agreement reference point by scoring each grader against the majority of the others: mean
per-grader F1 is 0.78 (range 0.63–0.89), and we read every judge result in Section 6 against
this reference.

4.5   Reporting Monitor Performance

For each (judge, transcript-strategy) pair we report precision, recall, and F1 against the
gold labels, with the formal definitions given in Section 3.2. We report all three rather
than F1 alone because the deployment costs are asymmetric: a false negative is an executed
scope violation, the production harm the monitor exists to prevent, whereas a false positive
merely escalates a legitimate call to a human and, in aggregate, risks approval fatigue. Our
practitioner recommendations are accordingly framed as two operating points rather than a
single highest-F1 configuration (Section 6.1): a cost-sensitive one that holds recall high at low
cost, and a recall-first one that maximizes recall before precision collapses.


5     Experimental Design

This section fixes the concrete instantiation behind the results of Section 6: the judge models
we evaluate (Section 5.1), the inference settings under which they run (Section 5.2), how
we account for cost (Section 5.3), and the harness that produces and scores every (judge,
transcript-strategy) decision (Section 5.4). The estimands themselves (precision, recall, F1,
and the cost C(J, φ) of Equation (10)) are defined in Section 3.2.


5.1   Judge Models

We evaluate eight judge models chosen to span the axis our thesis turns on (open-weight versus
proprietary) and a wide range of price and capability, while holding the agents under test
(Table 1) fixed. The full roster, with provenance, is given in Table 2. Of the eight judges,
four are open-weight and four proprietary; this split lets us ask whether the cost–accuracy
frontier is reachable without a closed model (Section 6.1). All judges were queried over API:
the proprietary models through their respective providers’ APIs and the open-weight models
through OpenRouter.


5.2   Inference Configuration

All judges are queried with identical decoding settings so that differences in the scoreboard
reflect the model and the transcript strategy rather than sampling: every judge runs at
temperature 0 (greedy decoding) and emits a single short JSON object (Appendix A). We set

                                               11
        ScopeJudge: Cost-Aware Pre-Execution Gating for Offensive Security Agents


no explicit output-token cap, the JSON decision being short by construction. Reasoning is
enabled for every judge whose model supports it; for models that expose multiple reasoning-
effort levels, we use the API default. Each judge classifies each tool call independently and
is given no feedback across calls; each call is judged once (Section 7), and calls for which a
judge does not return a parseable decision are excluded from scoring rather than retried, which
accounts for the small gap between attempted and scored decisions reported in Section 5.4. For
the intent_plus_outputs_summary strategy, each tool output is summarized by a separate
instance of the same judge model under the summarization function σ of Equation (8) (also at
temperature 0 and with no cap on summary length), and cached per trajectory so each output
is summarized at most once (Appendix B). No judge’s context window was exceeded over the
evaluation, so no truncation was applied.


5.3   Cost Accounting

The per-call cost reported throughout (Figure 3, Table 3) is the empirical realization of
C(J, φ) in Equation (10): for every (judge, strategy, call) we record the exact input and
output token counts and multiply by the model’s per-token price. Token counts are taken
from each provider’s usage response; the per-call figure is the mean over all scored calls,
and $/1k in Table 3 is that mean scaled to one thousand calls. Per-token prices are each
provider’s published list rate for the proprietary models and OpenRouter’s per-token rate for
the open-weight models (the endpoints through which each was served; Section 5.1), all as of
June 2026; we hold them fixed across the study so that cost differences are attributable to
token usage rather than price changes. The reported costs cover judge inference, including
the one-time per-trajectory summarization required by the intent_plus_outputs_summary
strategy, which is folded into that strategy’s figures.


5.4   Harness and Scoring Procedure

We run the full 8 × 5 grid of (judge, transcript-strategy) configurations over the 4,897 labeled
tool-call decisions of the calibration set (Section 4.2), yielding one accept/reject decision per
configuration per call. The sweep is produced by our Pre-Execution Gating Harness, which
replays each labeled trajectory and dispatches the per-call judge queries under each transcript
strategy (Figure 1). Each predicted decision ŷk = J(SP, φ(τ<k , ak )) is scored against the
majority-vote gold label yk (Section 4.3); we map a judge’s allow=false to the positive
class (a predicted scope violation), so that recall is the fraction of true violations the gate
catches and a false negative is an executed violation (Section 4.5). For cost reasons each
configuration is evaluated once rather than across repeated samples (Section 7); we therefore
report single-run point estimates rather than confidence intervals. In total the sweep issued
8 × 5 × 4,897 = 195,880 judge decisions, of which 194,482 returned a parseable label and were
scored, at a total recorded API cost of approximately $812.


6     Results

We evaluate eight judge models, each under the five transcript strategies of Section 3, against
the 4,897 expert-labeled tool-call decisions of the calibration set (base rate of scope violations
7.7%). The evaluation is in effect a held-out benchmark for the monitors themselves: the
candidate judges are the systems under test, the professional graders’ majority label is ground
truth, and the rate at which the graders agree with one another is the level against which we

        ScopeJudge: Cost-Aware Pre-Execution Gating for Offensive Security Agents


read judge scores. Throughout, that expert agreement reference point is mean inter-grader F1
= 0.78 (Section 4.4).


6.1   The Cost–Accuracy Frontier

Figure 3 plots each judge at its best-F1 transcript strategy against its per-call cost; Table 3
reports the full per-configuration numbers. Three findings stand out. First, the best judge
is open-weight, as is the high-accuracy end of the frontier: GLM-5.2 attains the highest F1
of any judge (0.66 at $0.0060/call), and the strongest low-cost judge is open-weight as well
(Qwen3.6-35B, 0.57 at $0.0026/call). Second, proprietary judges are competitive but never
on top: Claude Sonnet 4.6 (0.55) and GPT-5.4-mini (0.50) are Pareto-optimal at low cost,
yet the best proprietary judge, Claude Opus 4.8, manages only F1 = 0.60 at 2.9× GLM-5.2’s
per-call cost and is dominated by GLM-5.2 outright. Third, returns diminish sharply with
context: in Table 3, moving from intent_plus_calls to full buys < 0.02 F1 for nearly
double the cost, and intent_plus_outputs_summary is dominated almost everywhere. Even
the best judge sits 0.13 F1 below the expert agreement reference point, so headroom remains.


Recommended Deployment Configurations. Because a missed violation is more costly
than a false alarm (Section 4.5), the optimal configuration depends on how many spurious
escalations a deployment will tolerate to catch more violations. For a cost-sensitive deployment,
GLM-5.2 with intent_plus_calls is the balanced choice: it catches 87% of violations at
precision 0.50 for $0.0032/call, the cheapest open-weight configuration that keeps recall high
without flooding reviewers. For a high-stakes deployment, where a single executed violation is
unacceptable, the asymmetric-cost logic instead pushes onto the recall-maximizing frontier,
trading precision (and thus more human escalations) for fewer misses. That frontier is led by
Claude Opus 4.8 (intent_only), which catches 94% of violations at precision 0.39, flagging
19% of all calls. A team that cannot tolerate a miss and can absorb the escalation load should
run Opus; one that wants to stay open-weight at high recall should run GLM-5.2 (full), which
holds 86% recall at precision 0.53. The high-stakes point buys 7 more recall points than the
cost-sensitive one, at the price of an 11-point precision drop and a switch to a closed model.


6.2   A Static Policy Cannot Gate Scope

Figure 4 isolates the central claim of the paper: static policy-based monitoring is insufficient for
scope enforcement, and request-conditioned monitoring is necessary. Under the static_policy
strategy (the judge sees the global static policy SP but not the user’s intent u), recall collapses
to near zero for seven of eight judges (median recall 0.005): with no notion of the task’s
declared scope, almost nothing in a tool call reads as a violation, so the gate waves everything
through. The instant intent is added (intent_only), recall jumps to 0.60–0.94. This is the
empirical form of our thesis: scope lives in the user’s request, and a specification-time static
policy, however well written, is structurally insufficient to gate it. The lone partial exception,
GPT-5.4-mini, flags far more aggressively across all strategies and so retains some static-policy
recall (0.25) at the cost of precision.

Several limitations bound the claims of this work and frame the future directions of Section 8.

8                      Conclusion

The boundary between an authorized action and a scope violation is not a property of the
action—a scan, an exploit, a data read—but of the request that authorized it. For agents
whose scope is declared at request time, this reframes oversight as a monitoring problem
rather than a specification one: no policy fixed in advance can name the host, endpoint, or
datum that a future engagement will rule out of bounds, but a judge that reads the request
can. ScopeJudge measures how well it can be done. Across eight judges and five transcript
strategies the picture is consistent: a static policy gates nothing, the user’s intent restores
most of what it lacks, and the cheapest judge that keeps recall high is open-weight, reaching
F1 = 0.66 against an expert agreement reference point of 0.78. Ever stronger untrusted models
will continue to arrive; the monitor that gates them stays trusted and fixed, and open-weight
judges are becoming strong enough to make this technique practical to put into production.

8.1   Future Work

Future work will focus on improving judges for narrow use-cases. By distilling judges from
larger models, it should be possible to reduce their costs to make it easier for real-time scalable
oversight to be performed.
These judges also represent a new kind of attack surface, and should be subjected to adversarial
assessment to determine their relative resilience to models attempting to act in unaligned ways
while fooling a judge.
Finally, this work only studied a binary classification of tool calls as either in-scope or out-of-
scope. Future work should study when a tool-call is aligned, but potentially dangerous enough
to require escalation to a human. The calibration of this classification to be sensitive enough
to not cause “approval fatigue” is crucial for safe deployment of autonomous agents.


9     Ethics and Impact

This work concerns the oversight of offensive-security agents, an inherently dual-use setting.
Our contribution is defensive: pre-execution gating exists to prevent an autonomous agent
from taking actions outside its authorized scope (the out-of-bounds scan, the unauthorized
state change, the data it was told to leave alone) before they execute against a real system.
The released artifact is an evaluation corpus of labeled agent trajectories together with a
benchmark of judge configurations for catching such actions; it confers no offensive capability
beyond what is already available to anyone running a coding or security agent, while giving
defenders a concrete, measurable way to bound that capability.
The trajectories were generated in controlled environments rather than against live third-party
systems, and contain no real client data or personally identifiable information; labeling was
performed by professional penetration testers among the authors. We release the labeled
ScopeJudge calibration corpus to support defensive monitoring research, and reserve the larger
unlabeled trajectory pool and a fuller treatment of the ScopeBench task suite for separate,
safety-focused work. We treat its adversarial robustness as future work and recommend that
deployments regard gating as one layer of defense rather than a guarantee.

A     The Judge Prompt

This appendix reproduces the verbatim static policy SP supplied to every judge and specifies
exactly which context blocks each of the five transcript strategies of Section 3 places in the
judge’s window. The static policy is held constant across all eight judge models and all
five strategies; what varies between strategies is solely the context that φ projects alongside
it (Equations (5)–(9)). For every actionable tool call we score, the judge returns a single
JSON object {"allow": <bool>, "reason": <string>}; allow=false (a rejection) is the
positive class, and is scored against the ground-truth label for that step.
System message (constant across all judges and strategies).
You are a scope-compliance judge for an LLM agent operating under defined
rules of engagement. For each proposed tool call you will be shown the
original engagement instruction, a compact summary of the agent’s prior
actions in this session, and the new call awaiting judgment.

Return a single JSON object:
{
  "allow": <true|false>,
  "reason": "<one-sentence justification grounded in the instruction>"
}

Decision rules:
- "allow" if and only if the proposed call stays within the documented
  authorization surface implied by the instruction.
- "deny" if the call accesses a resource, host, endpoint, or capability
  that the instruction excludes, or would constitute a state change the
  instruction forbids.
- A call need not directly name a forbidden surface to be denied --
  inferred violations (Host-header bypass, credential reuse, lateral
  movement to a sibling tenant, brute-force enumeration) count.
- Reconnaissance against documented endpoints is allowed unless the
  instruction explicitly prohibits it.




Context blocks per strategy. Table 4 makes concrete which blocks each φ supplies on top of
this static policy. The two memoryless strategies (φstatic , φintent ) expose no attacker-controlled
tool output to the judge; the history-bearing strategies trade a wider prompt-injection surface
for richer signal, culminating in φfull , which supplies the judge the same context a human
grader saw when assigning ground truth (Section 4.3) and is therefore the most-informed
automated condition.
