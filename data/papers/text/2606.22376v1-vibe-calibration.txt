                                                         Superconducting quantum computing is one of the most mature solid-state platforms for quan-
                                                         tum computation, with processors exceeding one hundred qubits. Yet further scaling toward fault-
                                                         tolerant quantum computing is increasingly constrained by calibration complexity. Conventional
                                                         scripts are brittle to anomalous signals, and expert judgment is bounded by cognitive bandwidth
                                                         and serial operation time, failing to keep pace with system scale. Here we report Vibe Calibration,
                                                         an autonomous calibration system orchestrated by large language model agents, which distills expert
                                                         tacit knowledge into reusable Skills. Each Skill is organized as a decision tree that packages pa-
                                                         rameterized measurement commands, quantitative acceptance criteria, and audit records, enabling
                                                         autonomous execution and self-healing. We capture this knowledge through a three-phase human-
                                                         in-the-loop distillation process and fine-tune a large language model on validated trajectories. On
                                                         a 112-qubit processor with frequency-tunable transmons, the system autonomously completes cal-
                                                         ibration of 108 out of 112 qubits in 4.7 hours, achieving a 4–5× speedup over manual calibration
                                                         of the full 112 qubits. A cross-validated comparison with expert manual calibration on a 16-qubit
                                                         subset shows agreement on 14 out of 16 qubits. More importantly, the model demonstrates trans-
                                                         ferable calibration workflows across devices. While low-level control scripts require minor interface
                                                         adaptation for different hardware platforms, the core decision logic and task orchestration generalize
                                                         to new processors, demonstrating a reusable laboratory interface rather than a memorized script.
                                                         This work demonstrates, for the first time, fully autonomous calibration of a hundred-qubit super-
                                                         conducting processor through reusable and auditable Skills, removing a critical barrier to scalable
                                                         quantum hardware operation.

This Supporting Information describes an artificial-intelligence-assisted workflow for superconducting-qubit calibration and
characterization. The agentic system considered here has two coupled components: an agent scaffold that exposes laboratory
tools to large language models, and a collection of Skills that distill expert experimental experiences into executable workflows.
On the agent side, we tested Claude Code driven by fine-tuned Qwen3.6 models and KimiCode driven by Kimi K2.6. The
Qwen3.6 family includes several calibration-adapted checkpoints fine-tuned from real calibration data, whereas the Skills were
distilled from the operating procedures of experienced engineers rather than written as generic prompts. The following sections
first introduce the agent and model deployments, then describe the training-set construction and fine-tuning methodology, explain
how the qubit-characterization Skill is constructed and executed, report model performance and detailed outputs on concrete
quantum chips, and finally discuss the generality of the Skill interface beyond qubit characterization.


                                     I.   LARGE LANGUAGE MODELS AND THEIR DEPLOYMENTS

We evaluated and deployed three distinct large language models (LLMs) tailored for our system. The first is a fine-tuned
Qwen3.6-35B-A3B [1], a Mixture-of-Experts (MoE) model that activates 3B parameters per forward pass out of a 35B total.
The second is a fine-tuned dense 4B model [2], designed as a lightweight alternative for memory-constrained or cost-sensitive
deployment scenarios. The third setup uses Kimi K2.6 integrated with Kimicode [3–6].
We served the fine-tuned Qwen3.6 checkpoints behind the Claude Code command-line interface through a three-layer stack
consisting of vLLM for inference [7], a LiteLLM proxy translating between the Anthropic Messages API used by the client and
the OpenAI Chat Completions interface exposed by vLLM, and a custom pre-call middleware for context-window enforcement.
The vLLM process is bound to the loopback interface and the LiteLLM proxy is exposed as an authenticated public endpoint,
with authentication enforced at the LiteLLM layer through a master key supplied via environment variable. Because both base
models are multimodal while the supplied checkpoints contain only updated text-decoder weights, we performed a grafting
procedure in which tensors with key prefix visual.* are extracted from the original multimodal release into a dedicated
shard, the tokenizer and chat template are inherited from the fine-tuned release, the multimodal config.json is preserved, and
model.safetensors.index.json is regenerated to resolve text and visual keys to their respective shards. The grafted vision
tower remains inert during text-only inference because it is activated only when image tokens appear in the prompt. Clients
frequently request maximal output lengths in long multi-turn sessions, which causes the combined input and output budget to

                                              II.    TRAINING SET CONSTRUCTION

The training data are distilled from raw trajectories recorded while a Claude Code agent [8], driven by an LLM as its teacher
model and scaffolded by our skill library, performs calibration tasks under direct human supervision. Each raw trajectory
contains the agent’s tool calls and instrument responses interleaved with operator-issued clarifications, corrections, and on-the-fly
redirections. From this collection we extract two complementary training datasets, D 𝐴 and D 𝐵 , described below.


                                             A.     Dataset A: step-level behavior cloning

Dataset A teaches the model what action to take given a specific concrete experimental scenario, in the spirit of step-level behavior
cloning. In each source trajectory the agent typically passes through several interactive debugging and human guidance attempts
before reaching the step that finally satisfies the operator; only that operator-validated final step is retained as a training label,
paired with the ∼ 30 messages of preceding context that conditioned it: the operator’s instructions, the agent’s prior reasoning,
the tool calls it issued, and the instrument responses they returned.
Two preservation choices are critical for adapting the model to its target deployment environment. The full tool-schema list (the
definition of every tool available to the agent at the moment of decision) is carried in every sample, aligning the fine-tuning
interface with the action space the model will face at deployment. This design follows prior work showing that tool use can
be folded directly into the language-model training objective rather than left as a downstream prompting artifact [9]. At the
same time, we preserve the original Claude Code harness instructions verbatim in the system prompt. Since the target Qwen3.6
model [10] is not documented as being trained for the Claude Code agent wrapper, this step adapts the fine-tuning distribution to
that wrapper.


                                                    B.   Dataset B: domain knowledge

Dataset B teaches the model the core concepts, characteristic numerical regimes, and operational rules of the calibration domain.
From each source trajectory we mine a heterogeneous pool of candidate snippets, falling into six categories:

• numeric facts: concrete physical value ranges and expected magnitudes for equipment settings (e.g., typical 𝑇1 lifetimes or
  optimal readout amplitudes);
• domain rules: decision criteria, scaling relations, and judgment heuristics articulated during the session (e.g., “𝑅 2 < 0.9 means
  the fit is unreliable”);
• error-recovery sequences: a tool failure traceback paired with the reasoning and corrective retry that successfully resolved it;
• operator lessons: direct natural-language teaching turns, warnings, and explicit corrections issued by the human expert;
• raw tool outputs: verbatim instrument returns and console logs retained to expose the model to the exact text distributions it
  will face;
• working code snippets: validated Python blocks and tool-call payloads that successfully exercised the in-house quark/measlib
  APIs.

A single DeepSeek-V4 [11] pass then filters this pool, casts each surviving snippet into a typed atom with a one-sentence canonical
statement, and normalizes the session-specific identifiers (concrete run identifiers, operator accounts, and absolute filesystem
paths) into role-preserving placeholders, so that the model learns the underlying role of each identifier rather than the contingent
value it happened to carry in one particular session.
The atoms are then expanded into surface-form variation. Because factual knowledge is easier to store and extract when it
appears under diverse formulations [12], the same DeepSeek-V4 model rewrites each atom into 6–8 paraphrases, following
B     Dataset B: domain knowledge                                                                                                                                   3

the motivation of paraphrase-based data augmentation [13] drawn from a fixed bank of templates: direct narration, question-
and-answer, counter-factual inversion, story-of-past-incident, simulated operator–agent dialogue, and critique of a deliberately
incorrect claim. By decoupling facts from specific phrasings, this structural diversity forces the model to encode the semantic
invariant rather than memorizing surface syntax.

                                               III.        FINE-TUNING METHODOLOGY AND SETUP

We fine-tuned two base models, Qwen3.6-35B-A3B and Qwen3.5-4B, on the datasets described in Section II using LoRA-
Over [14], a parameter-efficient fine-tuning (PEFT) framework that improves generalization over vanilla LoRA by over-
parameterizing the low-rank adaptation matrices during training. LoRA-Over applies matrix decomposition to enable near-
lossless reconstruction of the effective adaptation, so that the over-parameterized factors can be absorbed back into standard
low-rank form at inference time with no additional cost. We describe the fine-tuning setup in two parts. Section III A details the
training procedure, including dataset splits, hyperparameters, and MPO architecture configurations. Section III B introduces the
two base models and their architectural properties relevant to the adaptation.



                                                                      A.    Training procedure

run optimizes for step-level behavioral cloning, teaching the model which concrete tool call to emit given a full conversational
context and tool-schema list. The D 𝐵 run optimizes for calibration domain knowledge, covering numeric facts, domain rules,
and operational heuristics. The training configurations for all four runs are summarized in Table II, and the corresponding MPO
architectures are listed in Table III. The evolution of the validation loss over training for each run is shown in Figure 1.

                                                                             B.     Models

a. Qwen3.6-35B-A3B. This mixture-of-experts model activates 3B parameters per forward pass out of 35B total. The large
parameter count provides representational capacity for the calibration domain, while sparse activation keeps inference latency
comparable to a dense 3B model. LoRA-Over adapters are attached to all attention projection matrices and feed-forward gate
projections in each expert.
b. Qwen3.5-4B. This dense 4B model serves as a lightweight alternative intended for deployments where memory footprint
or inference cost is constrained. Its more limited capacity makes domain-knowledge acquisition from D 𝐵 especially important
for reliable performance.
c. KimiCode with Kimi K2.6. We also ran a negative control with KimiCode backed by Kimi K2.6 on the q112 sweet-spot
Skill. The exported session contained only 3 user turns but expanded into 69 tool calls and 62k tokens, with 41 separate thinking
blocks. Qualitatively, the model repeatedly re-analyzed the skill, environment, and possible interpretations of the command,
but did not reliably convert this reasoning into the requested experimental action. In one representative smoke-test request, it
spent many steps checking files, reconsidering SSH topology, and debating whether to run or merely report; when it finally
attempted the scan, the foreground timeout killed the command, after which it continued reasoning about alternatives rather than
cleanly switching to a robust background execution strategy. This failure mode is distinct from the Qwen fine-tuning failures
above: it is not primarily old-skill memorization, but an execution-control problem in which excessive deliberation and weak
action commitment prevent completion of a time-sensitive laboratory task. We therefore did not use KimiCode/K2.6 as a main
evaluation model.


                              IV.   CONSTRUCTION AND EXECUTION OF THE QUBIT-CHARACTERIZATION SKILL

The evaluated calibration task was not exposed to the model as an informal natural-language instruction. We first converted the
laboratory procedure into a Claude-Code Skill, following Anthropic’s Agent Skills pattern in which a skill is a filesystem-based
package of instructions, scripts, and resources that can be discovered and loaded on demand [15, 16]. The Skill makes the
                                                                                                                                       5

experimental workflow executable by an agent while keeping the physical control routines unchanged. It wraps the existing
qulab/quarkstudio calibration scripts with four explicit layers: an orchestration layer that decides which measurement should
be run next, a script-interface layer that presents every measurement as a parameterized command, a gate layer that converts fits
and physical checks into pass/fail decisions, and an audit layer that records the result identifiers, figures, raw arrays, fitted values,
and follow-up decisions produced during the run. This design separates the problem of operating the calibration workflow from
the problem of writing new low-level measurement code.
The construction process followed the same pattern for each calibration node. First, we identified the human-readable experimental
intent, such as finding the readout resonator, selecting a candidate qubit frequency, validating that the candidate is the 0 → 1
transition, or refining the frequency by Ramsey. Second, we exposed the corresponding control routine as a command with
explicit arguments, including qubit list, sweep range, point count, drive amplitude, pulse width, signal type, and a short motivation
string. Third, we defined the expected artifacts of the command: a result identifier, one or more plots, numerical arrays, fitted
quantities, and configuration values that may be written back. Finally, we wrote a gate that states when the result is accepted,
when the agent should retry with a modified sweep, when the qubit should be removed from downstream batches, and when the
workflow must stop for human inspection.
Concretely, the Skill consists of five components, summarized in Table IV. The orchestrator stores the decision tree and chooses the
next calibration node. The script interface exposes each measurement as a command with explicit arguments. The configuration
adapter maps conceptual quantities into the schema used by the target chip and control stack. The decision gates prevent weak
peaks, aliased Rabi oscillations, unstable readout settings, or divergent Ramsey corrections from silently propagating downstream.
The audit layer preserves the intent, parameters, result identifiers, fitted values, figures, raw arrays, and next action for each run.

The resulting workflow is a decision tree rather than a flat script, as shown in Fig. 4. A typical run begins by initializing the
trigger and acquisition state, optionally validating the ADC trace, and running S21 to locate the readout cavity. The agent
then performs qubit spectroscopy to select a candidate 𝑓01 , validates the candidate with time Rabi at multiple drive amplitudes,
calibrates the 𝜋 pulse with power Rabi, performs single-shot and readout-amplitude optimization when available, measures 𝑇1 ,
and finally runs Ramsey to refine the qubit frequency. Optional branches include 𝑇2R and higher-transition checks after the base
frequency has converged. Optional flux-tunable multi-qubit devices add a separate flux-arrangement route before entering the
shared characterization chain; the two routes are shown separately in Fig. 2. The agent is not allowed to treat later steps as
independent: every downstream node consumes state written by earlier nodes, such as Measure.frequency, R.frequency,
R.amp, R.width, threshold, rotation angle, and signal-type choice.

Multi-qubit execution required an additional dispatch rule. The experimental groups are not inferred from Rabi spectra; they are
taken from the device topology exposed by the toolbox backend. In the S8 execution this produced four topology-defined groups
of 28 qubits, which are the units submitted to the Ctoolbox characterization routines. The Skill uses these topology groups as
the hardware-safe batching boundary, then filters qubits within each group by the validity of upstream calibration state before
dispatching downstream measurements. This distinction matters because grouping is a hardware-topology constraint, while fit
quality and sweep resolution are later per-qubit gates.


For flux-tunable multi-qubit devices, the workflow also includes a flux-arrangement branch. Before running the single-qubit chain
at scale, the agent runs S21-vs-flux to locate flux-dependent readout shifts and spectrum-vs-flux to map the qubit transition curves.
The spectrum-vs-flux data are harder to interpret than one-dimensional spectra because multiple near-parallel ridges, spectator
lines, and two-photon features can coexist in the same image. We therefore use a data-driven ridge extractor rather than visual
selection. Each flux slice is first background-normalized, local peak candidates are extracted, and dynamic programming is used
to recover several smooth ridge paths through the two-dimensional image. Each path is fit to a quadratic or transmon-like curve,
low-quality paths are rejected by 𝑅 2 , RMSE, ridge span, and curvature gates, and the highest-frequency member of the remaining
near-parallel family is selected as the one-photon branch. This implements the empirical rule that, for negative-anharmonicity
transmons, the one-photon branch is the upper branch when a lower two-photon-like ridge is also visible [17–19].

                        V.   PERFORMANCE OF THE FINE-TUNED MODELS ON A 16-QUBIT CHIP (S4)

We fine-tuned several Qwen-based models on calibration skill data (qubit-characterize) and deployed them as independent
service instances. The evaluated models span two base architectures: the 35B series, built on Qwen3.6-35B-A3B with a
Mixture-of-Experts (MoE) design (35B total, ∼3B active parameters), and the 4B series, built on Qwen3.5-4B with a dense
architecture (4B parameters). To make the training-set provenance explicit, this section names each checkpoint by its base model
and fine-tuning dataset: 35B-D 𝐴, 35B-D 𝐵 , 4B-D 𝐴, and 4B-D 𝐵 . This section evaluates the generalization capability of these
fine-tuned models when presented with a previously unseen skill targeting a different chip architecture, while controlling for
base-model capability as a confounding factor.
In terms of experimental design, all models were fine-tuned on a single skill (qubit-characterize) targeting a different chip,
which differs systematically from the test skill in workflow structure, chip topology, and execution interface. At test time, we
present the models with a new skill (s4-cali-babystep) for a 16-qubit chip absent from the training data. The sole evaluation
metric is skill adherence: whether the model correctly reads and follows the new skill, or regresses to patterns memorized during
fine-tuning. As shown in Table VII, the four models exhibit a systematic gradient from successful transfer to severe overfitting.

At the behavioral level, 35B-D 𝐴 exhibits three salient characteristics: across all 6 sessions the model never proactively read or
referenced the training skill’s documentation; regardless of the loading mechanism, whether system trigger, file read, or manual
load, the model consistently followed the new skill; and all protocol details prescribed by the new skill were correctly executed.
The results of 35B-D 𝐴 indicate that, given sufficient base-model capability, fine-tuning can simultaneously achieve effective
domain-knowledge transfer and preservation of instruction-following capacity. However, the contrasting behavior of 35B-D 𝐵
(discussed below), which shares the same base architecture, suggests the existence of a critical moderating variable during fine-
tuning, such as data volume, data quality, or training strategy, that determines whether adaptability is retained or lost. Identifying
this variable has important implications for future model training.


                           B.   35B-D 𝐵 : Domain Knowledge Transfer with Compromised Skill Switching

35B-D 𝐵 exhibits inconsistent transfer behavior across 5 test sessions: the fine-tuning prior manifests in all sessions, with severity
ranging from complete disregard of the new skill to subtle file-directory memorization. In a representative case, the new skill
was successfully injected into the context, yet the model entirely disregarded it, reverting to the training skill’s documentation
and executing according to its patterns. This indicates that the prior distribution established through fine-tuning is sufficiently
strong to override explicitly provided new information. Another manifestation is file-structure memorization: in certain sessions,
the model attempted to read sub-documents following the training skill’s directory layout, files that do not exist in the new skill.
It persisted despite repeated “File does not exist” errors, suggesting that fine-tuning encoded not only workflow content but also
file organization patterns. Context compaction serves as an additional trigger: in some sessions regression to the training skill’s
documentation was observed after compaction, while in others the model mentioned old-skill-specific steps before compaction
occurs (later self-corrected).
The infiltration of the fine-tuning prior is observed not only at the macro-behavioral level but also in specific parameter choices.
In certain sessions, the model used 2 amp-duration pairs for TimeRabi, matching the training skill’s default configuration
rather than the new skill’s prescribed 3 q112-style pairs (0.10/500ns, 0.20/250ns, 0.30/180ns). This reveals prior leakage at the
parameter level: the model does not deviate randomly from the new skill’s specification but precisely reproduces the training
skill’s parameter defaults. Following human correction, the model re-read the new skill and re-executed with the correct 3-group
parameters.
A further behavior was observed in some sessions: after correctly completing the full workflow, the model received a request for
a step absent from both skills. Rather than flagging this instruction as outside the current skill’s scope, the model acknowledged
and prepared to execute it. While it subsequently verified the script’s existence and correctly reported its absence, when pressed
it generated a confabulated claim of having previously implemented the step.

                                             C.   4B-D 𝐴 and 4B-D 𝐵 : Transfer Failure

Both 4B-series models failed the transfer task, though their failure modes differ qualitatively. 4B-D 𝐴 exhibits complete
memorization: the model failed to load the new skill, and its output constitutes a verbatim reproduction of the training skill’s
workflow, including steps unique to that skill. The model demonstrates complete memorization of the fine-tuning content with
no ability to ingest new instructions. In contrast, 4B-D 𝐵 has a more fundamental failure mode: the model cannot locate skill
files, construct valid commands, or produce coherent output, instead generating confabulated content referencing scripts and
procedures that exist in neither skill. This failure is not attributable to skill switching per se but reflects a deeper capability
degradation: fine-tuning appears to have exacerbated the inherent limitations of the 4B dense base model in tool use and
reasoning, resulting in the loss of basic competencies such as instruction parsing, file location, and error recovery, replaced by
the generation of plausible-sounding but fictitious content.


                                                           D.   Discussion

These results identify two distinct manifestations of fine-tuning overfitting. The first is pattern lock-in (35B-D 𝐵 and 4B-D 𝐴),
wherein the model memorizes specific execution patterns from the training skill and tends to reproduce these learned patterns
rather than adapting to new content; 35B-D 𝐵 exhibits intermittent pattern regression while 4B-D 𝐴 exhibits complete pattern
lock-in. The second is capability degradation (4B-D 𝐵 ), where the fine-tuning process amplifies the base model’s inherent
D   Discussion                                                                                                                 10

limitations in tool use and reasoning, resulting in the loss of fundamental competencies and the production of confabulated
output.
It should be noted that the 35B series (MoE, 35B total/∼3B active) and the 4B series (dense 4B) differ fundamentally in parameter
count and architecture, constituting a significant confounding factor. The 4B series’ failure should therefore not be attributed
solely to overfitting: its base model may inherently lack the tool-use and instruction-following capacity required for complex
skill-switching tasks. In contrast, the comparison between 35B-D 𝐴 and 35B-D 𝐵 offers higher internal validity: both share the
same base architecture, so their performance gap can be more directly attributed to differences in the fine-tuning process.
Overall, the successful transfer of 35B-D 𝐴 demonstrates that, given sufficient base-model capability, fine-tuning can achieve
both domain-knowledge transfer and instruction-following preservation. For calibration scenarios requiring cross-chip transfer,
models exhibiting similar characteristics should be prioritized; base-model selection is equally critical, as smaller dense models
may lack the capacity to support complex skill switching. For models exhibiting pattern lock-in (e.g., 35B-D 𝐵 ), stronger
contextual guidance at inference time (e.g., explicitly suppressing references to the old skill in the system prompt) may provide
partial mitigation. It is worth noting, however, that for deployments serving a single chip without cross-chip generalization
requirements, mild overfitting, i.e. partial memorization of the training skill, need not be a defect: such models may achieve
greater execution stability and consistency under a fixed workflow than models with stronger generalization, representing a
reasonable engineering trade-off.


                                VI.   GENERALITY BEYOND QUBIT CHARACTERIZATION

The Skill interface was not limited to the qubit-characterization workflow used for model training and evaluation. We also
built a separate tl-characterize Skill for transmission-line characterization, motivated by an immediate laboratory need
rather than by the training set. This task required identifying readout-resonator features along each transmission line, checking
power-dependent Kerr shifts as evidence of coupling, and producing records that could be inspected before downstream qubit
calibration [18–20]. The procedure was derived from an engineer’s notebook but exposed to the agent as a parameterized
experimental interface.
The resulting Skill acts like a domain-specific command-line interface. For example, an invocation such as /tl-characterize
11 7 specifies transmission-line channel 11 and seven expected devices; the agent expands this compact command into a short
closed workflow rather than a single scan. The loop is shown in Fig. 4. The agent first runs a wide overview scan, narrows the
frequency window, runs fine scans at two powers, checks whether the detected peaks match the expected device count, compares
the two powers to identify Kerr shifts, and then runs an S21-vs-power map before submitting a summary [21, 22]. When the
peak count or matching is inconsistent, the workflow returns to the fine-scan stage with adjusted range, power, or peak-detection
settings. This is the part that would otherwise live in a human-operated notebook as repeated visual inspection, parameter tuning,
and figure export.
Figure 5 shows the corresponding artifacts for a representative TL16 run. In the campaign, 14 available transmission lines were
processed, corresponding to 98 designed qubit sites. We do not use this result to claim a precise human-time speedup, since an
experienced engineer focusing on the same task could also make the notebook workflow efficient. The important point is different:
a non-expert operator can invoke the Skill and obtain the kind of focused, auditable output that would normally require sustained
expert attention. This demonstrates that Skills are not merely memorized task scripts for one calibration benchmark; they are
reusable laboratory interfaces for turning new experimental procedures into executable agent workflows without retraining the
model.

                                                      B.   Agent-level comparison

The second part of NVIDIA’s release is the Quantum-Calibration-Agent Blueprint [25], an open-source reference framework
that exposes quantum calibration experiments as LangChain tools to a general-purpose LLM agent. Its architecture includes
AST-based experiment discovery, subprocess-isolated execution, dual HDF5/SQLite storage, and a DAG-based workflow engine.
The agent’s behavior is guided by a three-layer knowledge architecture (Documents/Skills/Memory): a system prompt defining
the agent role, six skill directories prescribing abstract execution procedures (e.g., a 7-step experiment execution loop, a 4-step
workflow planning protocol), and a reference documents layer intended for domain knowledge.
In the public release (as of May 2026), the skills contain meta-procedures—rules for how to plan, execute, and retry experiments—
but not the concrete calibration content such as experiment sequences, parameter selection strategies, quantitative quality gates,
or decision trees for failure handling; these decisions are instead delegated to the underlying general-purpose LLM at run time.
The Documents layer, intended for domain-specific references such as parameter ranges and troubleshooting guides, contains
only an empty .gitkeep placeholder. All bundled experiment scripts generate simulated data; no real-hardware interfaces
are included. Because the publicly released artifacts do not contain the domain knowledge or hardware interfaces required for
real-machine calibration, we do not treat the QCA blueprint as an experimental baseline.
For context, in our system the calibration domain knowledge is encoded directly in structured Skills. The qubit-characterization
Skill contains a complete decision tree (Fig. 4) that orchestrates nine parameterized sub-skills, each with quantitative quality gates
and explicit failure-handling branches (Table IV), distilled from the operating procedures of experienced engineers as described
in Section II. The Skills are authored once and invoked without modification across devices, as demonstrated by the cross-chip
transfer experiment.


                                                           VIII.   SUMMARY

These results show how superconducting-qubit calibration can be organized as an agentic laboratory workflow rather than as an
isolated language-model benchmark. The deployment stack connects large language models to the Claude Code or KimiCode
agent interfaces; the training data transfer both step-level tool-use behavior and calibration-domain knowledge; and the fine-tuning
procedure adapts Qwen-family models while preserving, to varying degrees, their ability to follow newly supplied Skills. The
qubit-characterization Skill demonstrates how expert calibration practice can be distilled into a structured, executable protocol
with explicit quality gates, retry logic, and audit artifacts. The chip-level evaluations further show that model capacity and
training details strongly affect whether a fine-tuned agent follows a new Skill or regresses to memorized calibration patterns.
Finally, the transmission-line characterization example indicates that the same Skill abstraction can package new laboratory
                                                                                     13

procedures into reusable agent interfaces without retraining the underlying model.
