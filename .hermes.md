# skill-bench Project Rules

## Canonical direction

**Read `PROJECT_CHARTER.md` before choosing or executing work.** It is the repository's master statement of founding intent, scope, objectives, success criteria, and anti-drift boundaries. If this file, a queue task, a report, or another document conflicts with the charter, the charter wins.

The benchmark is intentionally about **knowledge work across domains**. Do not silently narrow it to one profession, workflow, artifact type, pilot, or use case. A narrow pilot is a bounded experiment: state the general hypothesis it tests and the reusable knowledge or machinery it should produce.

Before adding or claiming a task, apply the charter's decision filter:

1. name the charter objective advanced;
2. identify the concrete evidence or artifact produced;
3. state the uncertainty or capability it clarifies;
4. classify it as expansion, consolidation, building, validation, or human learning;
5. check for duplication and accidental scope narrowing;
6. define what useful completion looks like.

The founding intent in charter §§1–5 changes only with Samuel's explicit direction. Workers may propose evidence-backed updates to working principles, strategy, links, and status as the repository evolves. The consolidator must periodically check charter/repository alignment and adapt subordinate docs without rewriting the founding intent. It must also maintain `docs/research-synthesis-index.md` as the human-facing map of grouped insights, source relevance tiers, cross-paper conclusions, and direct benchmark implications.

## Mission

Build a compounding research and implementation system for converting tacit domain expertise into highly useful benchmarks of agentic knowledge work.

Every autonomous run must advance at least one mission outcome:

1. deepen understanding of relevant research or production systems;
2. convert learning into durable benchmark-design knowledge;
3. build/test benchmark infrastructure, schemas, tasks, rubrics, or pilot scenarios;
4. consolidate and improve existing project artifacts;
5. identify a strategic decision that requires Samuel's input.

Do not optimize Hermes for its own sake. Process improvements are valuable only when they improve research quality, benchmark construction, or Samuel's understanding.

## Operating model

The repository has two fronts:

- **Expansion:** discover, download, read, and review relevant sources.
- **Consolidation/building:** turn accumulated learning into benchmark artifacts and remove duplication/drift.

Use `data/work_queue.json` as the shared task queue. Before inventing work, inspect the queue and recent commits. Claim at most one task per run. Avoid duplicating work already claimed or completed.

## Evidence standard

- Never claim to have read a paper unless its PDF/full text was actually downloaded or fetched and read.
- Record source URLs and local PDF/text paths when available.
- Abstract-only summaries must be explicitly labeled as triage, not deep reviews.
- Reviews must explain unique insight, methodology, limitations, transfer to skill-bench, and concrete actions.
- Prefer primary sources: papers, official engineering posts, repos, documentation, and evaluation artifacts.

## Benchmark-building standard

Translate domain expertise into explicit primitives:

- hidden requirements;
- trusted evidence and provenance;
- contradictions;
- decision thresholds;
- artifact conventions;
- failure signatures;
- rubric checks;
- safety/compliance constraints.

Build artifacts, not just prose. Valuable outputs include schemas, validators, source packs, task specs, rubrics, graders, pilot tasks, tests, and diagnostic reports.

## Self-improvement protocol

The self-improvement system is a means for achieving `PROJECT_CHARTER.md`, not a separate project and not evidence that the benchmark itself must focus on self-evolving agents. Research on self-improving agents is relevant only when it improves the research/build organization or informs a legitimate benchmark question.

Treat process changes as experiments:

1. state a hypothesis;
2. make a bounded change;
3. define observable evidence;
4. record outcome in `docs/self-improvement-ledger.md`;
5. keep, modify, or revert.

Do not let cron workers create, remove, or reschedule cron jobs. They may propose schedule changes for Samuel.

## Autonomy levels

### Level 0 — safe autonomous

- source discovery/metadata;
- paper download/extraction;
- paper reviews;
- daily/scouting reports;
- small documentation fixes;
- queue updates.

### Level 1 — autonomous, then report

- templates, schemas, validators, scripts, tests;
- pilot benchmark drafts;
- non-destructive repo organization;
- small verified implementation changes.

### Level 2 — ask Samuel first

- external outreach or sponsor contact;
- public claims/releases;
- repo visibility changes;
- paid services or intentional spend;
- deleting major artifacts;
- major architecture or cron changes.

## Git discipline

- Pull with `git pull --ff-only` before edits.
- Commit only meaningful verified changes.
- Use role-prefixed messages: `scout:`, `review:`, `build:`, `synth:`, `fix:`.
- Direct small verified commits to `main` are allowed. Use a branch for large/risky changes.
- Never force-push or rewrite history.

## Verification

Before a code push, run relevant checks. At minimum:

```bash
python -m py_compile scripts/*.py
python scripts/check_review_quality.py papers --allow-empty
python scripts/queue.py validate
```

## Human communication

Workers run quietly. Human-facing briefs should be concise and decision-relevant:

- what was learned/built;
- evidence and links;
- effect on benchmark direction;
- decisions needed from Samuel.
